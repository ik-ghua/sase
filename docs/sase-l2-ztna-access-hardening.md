## 一、背景

我们的 ZTNA 已实现**应用零入站暴露**(App Connector 出站反向,L1 3.8)——但那保护的是"应用",**PoP 接入面本身必须监听**(Agent/CPE 要连 PoP 建隧道),故 **PoP 的接入端口对公网可被扫描/探测/DDoS**(L1 3.12 攻击面)。零信任的"网络隐身/最小暴露"原则要求:资源对未授权方**不响应任何包**。→ 补 **SPA 单包授权**隐身 PoP 接入面。

撤销侧:我们有**短 TTL 凭证(分钟级到期)+ 吊销表(推到 PoP、数据面切流)**,但这两者**不直接指挥终端**——无法秒级让 Agent 强制重认证、按需重采姿态、或即时推配置。零信任"持续验证"要闭环到端。→ 补**终端实时控制通道**。

两项都是通用零信任标准件,本文基于公开原则设计,适配我们的栈(XDP/多租户/国密敏捷)。

---

## 二、目标(可衡量)

1. **PoP 接入面隐身**:PoP 对接入端口**默认丢包**,仅对通过 SPA 敲门验证的源短时开放;扫描者**探测不到**(无响应)。
2. **SPA 多租户 + 加密绑定**:敲门携设备/租户身份(加密绑定),授权后进对应租户域;敲门加密算法**敏捷(待国密 PoC-G)**。
3. **终端实时通道**:控制面/风险引擎可**秒级**推送 撤销/强制重认证/按需重采姿态/配置更新到 Agent;Agent 事件驱动上报姿态/风险信号。
4. **不重复、不越界**:实时通道**不改变"PoP PEP 权威、客户端不可信"**(L1 3.8);只提速信号与 UX,非新安全边界;不做 EDR/DLP 本体。
5. **抗 SPA 自身被滥用**:敲门洪水/重放/扫描有防护(XDP 廉价预过滤 + 限速 + nonce/时间戳)。

**非目标:** 加密算法(PoC-G);PEP/数据面实现(PoP L2);EDR/DLP 终端管控本体(P5);字节 schema;编码。

## 三、设计

### 3.1 SPA 单包授权(PoP 接入面网络隐身)

**背景** PoP 接入端口必须可被 Agent/CPE 连接,但不该被任意公网方探测到。SPA(Single Packet Authorization)是公开的"先认证、再开放"网络隐身技法。

**目标** 定 SPA 协议要点与隐身语义。

**设计 —— SPA 流程**
1. **默认隐身**:PoP 对接入端口(WireGuard/TLCP UDP/TCP)**默认丢包**(无任何响应)——扫描者看不到开放端口(L1 3.12 最小暴露)。
2. **敲门**:Agent/CPE 连隧道前,先发**单个认证 UDP 包(SPA knock)**到 PoP 敲门端口,含:**设备身份(设备证书指纹/公钥,L1 3.5)+ tenant_id + 时间戳 + nonce**,**签名/加密**(算法敏捷:国密 SM2/SM4 或标准,待 PoC-G)。
3. **验证**:PoP 验签(用该设备已注册公钥/证书)+ 时间戳窗口(±数十秒)+ nonce 防重放 + 限速。
4. **开放**:验证通过 → PoP 把**该源 IP(可含端口)写入短 TTL 白名单** → 仅对该源开放隧道端口;Agent 随即建隧道。
5. **过期**:白名单条目短 TTL;隧道建立/心跳续期,否则到期自动撤销(恢复隐身)。

依据:SPA 把 PoP 接入面从"对全网可探测"变为"仅对已认证设备短时可达",显著缩小攻击面/抗扫描/抗 DDoS(L1 3.12);敲门携设备身份 + 加密绑定,与我们 WG 公钥绑定哲学一致(L1 3.2)。
- 备选:不做 SPA,仅靠隧道层认证。落选:接入端口仍对全网暴露、可被扫描/DDoS/漏洞探测;隧道认证只挡未授权连接、不挡"被发现"。
- 备选:用应用层鉴权替代。落选:应用层之前包已到达,未实现"网络层不响应"的隐身。

**风险** 敲门包本身可被 DDoS(洪水无效敲门)→ XDP 廉价预过滤 + 限速(3.2);时钟偏移 → 窗口容忍 + 端侧时钟同步;首连多一次往返 → 延迟增量小、可接受(对接 L1 3.19,非每包)。

**结论** PoP 接入端口默认丢包(隐身),Agent/CPE 先发**加密签名的单包敲门**(设备身份+租户+时间戳+nonce),验证通过短时开放该源;缩小攻击面、抗扫描/DDoS;加密敏捷待国密。

### 3.2 SPA 落地:XDP 默认丢包 + 用户态验签 + 白名单

**背景** 我们数据面已有 **XDP 早丢**(L1 3.7 / PoP L2 3.2),SPA 应贴这套 eBPF 栈,而非引入 iptables/ipset。但 eBPF 难做非对称验签(SM2/签名)。

**目标** 定 SPA 在 XDP+用户态的职责切分。

**设计 —— 两层(XDP 快路径 + 用户态验签)**
- **XDP(内核快路径)**:接入端口**默认丢包**(allow-map 未命中即 drop);维护 **per-source 白名单 map**(SPA 授权后写入,带 TTL);对敲门端口做**廉价预过滤**(包长/格式/限速)后,把候选敲门包**上送用户态**(XDP_PASS 到 SPA 验证器,或 AF_XDP/perf buffer)。
- **用户态 SPA 验证器(PoP-agent 内)**:做**密码学验签**(SM2/标准,算法敏捷,eBPF 做不了)+ 时间戳/nonce/限速 → 通过则**写 XDP 白名单 map**(源 IP + TTL)。
- **白名单 TTL + 续期**:短 TTL;隧道建立后由数据面续期,断开/超时自动移除(恢复隐身)。
- **多租户**:验证器从设备证书/公钥解出 `tenant_id`,授权后该源进对应租户路由域(PoP L2 3.3 隔离);白名单可带租户标记。

依据:XDP 默认丢包 + 廉价预过滤扛洪水(快、省 CPU),crypto 验签放用户态(eBPF 不支持非对称);复用我们既有 XDP 栈,不引入 iptables/ipset(贴栈、性能)。
- 备选:全程 iptables/ipset(传统 SPA 做法)。落选:不贴我们 eBPF 数据面栈、性能与一致性差。
- 备选:在 eBPF 里做验签。落选:eBPF 难做 SM2/非对称验签;故 crypto 留用户态。

**风险** 上送用户态的敲门包成放大点 → XDP 限速 + 预过滤先扛;白名单 map 规模 → TTL 自动回收 + 上限;XDP 对网卡/驱动要求 → 与数据面 XDP 同硬件要求(L1 M1)。

**结论** XDP 默认丢包 + per-source 白名单 map + 廉价预过滤扛洪水;用户态 SPA 验证器做验签(算法敏捷)并写白名单;TTL 自动恢复隐身;贴我们 eBPF 栈、不用 iptables。

### 3.3 终端实时控制通道(持续验证到端)

**背景** 短 TTL(分钟级)+ 吊销表(到 PoP)不能秒级指挥**终端**(强制重认证/按需重采姿态/即时推配置)。零信任"持续验证"需闭环到端。

**目标** 定 Agent↔控制面的实时通道与其承载。

**设计**
- **通道**:Agent 维持到控制面的**持久控制连接**(复用我们 Agent↔控制面 mTLS 控制通道,加**服务端推送**);形态候选 **gRPC 双向流 / WebSocket**(选型待确认,附录)。按设备/租户身份鉴权(mTLS + 凭证)。
  > 与 xDS server 的区别:本通道是 **Agent↔控制面**;xDS server 是 **PoP↔控制面**(配置下发)。两者都是海量长连,但**承载/schema 不同、不复用同一 xDS server**(规模化思路可类比,见 RH4)。
- **服务端 → 端(推送)**:
  - **撤销 / 强制重认证**(立即让会话失效、要求重登);
  - **按需重采姿态**(风险引擎怀疑时,要 Agent 立刻上报新姿态);
  - **降权提示 / 重新评估**(强制端重走访问决策——**判定仍在 PoP PEP**);
  - **配置更新**(split-tunnel/内部域名/PoP 列表变更,推而非轮询);
  - **重新选路**(PoP 变更 → 重连)。
- **端 → 服务端(上报)**:心跳(设备存活)、**事件驱动**姿态/风险信号上报(非仅周期)、隧道状态。
- **存活管理**:Redis 维护设备在线态(短 TTL 心跳),供精准定向推送;离线则推送转入待发/下次上线。

依据:实时通道把"持续验证"从"被动等凭证过期"变成"主动秒级推送",支撑按需重采姿态与即时撤销 UX;复用既有控制通道 + 加推送,增量小。
- 备选:仅轮询。落选:撤销/重采姿态延迟到轮询间隔,慢;无法按需即时触发。
- 备选:仅靠短 TTL 凭证过期。落选:分钟级延迟、且不能按需重采姿态/推配置。

**风险** 海量持久连接规模 → 水平扩展 + 心跳(类 xDS server 思路);通道断 ≠ 安全失效(见 3.4 分工);客户端被控 → 推送对端是"建议/触发",**权威切流在 PoP**(3.4)。

**结论** Agent 持久控制通道(复用 mTLS + 服务端推送),秒级推 撤销/重认证/重采姿态/配置/重选路,端事件驱动上报信号;Redis 管存活定向推送;通道是提速非安全边界(3.4)。

### 3.4 与既有撤销/持续验证机制的分工(不重复、PoP 权威)

**背景** 我们已有 短 TTL 凭证、吊销表(到 PoP 数据面切流)、凭证刷新——实时通道不能与之重复或越过 PoP 权威。

**目标** 厘清三者分工,确立**权威在 PoP、实时通道提速、短 TTL 兜底**。

**设计 —— 撤销/持续验证的三层(各司其职)**

| 机制 | 作用 | 权威性 | 时延 |
|------|------|--------|------|
| **吊销表 → PoP PEP**(已有,xDS 独立流) | 数据面**权威切流**(拒绝已撤会话/凭证) | **权威**(强制点) | 可达时秒级 |
| **终端实时通道 → Agent**(本文 3.3) | 端侧**提速 + UX**:强制重认证/重采姿态/即时反映 | 非权威(端不可信,只是触发) | 秒级 |
| **短 TTL 凭证到期**(已有) | **不可达兜底**:网络断时凭证自然失效 | 兜底 | 分钟级 |

- **关键原则**:**权威切流永远在 PoP PEP**(客户端不可信,L1 3.8);实时通道**不是安全边界**——它让端"更快知道并配合"(重认证/重采),但即使端不配合或通道断,**PoP 吊销表 + 短 TTL 仍强制**。
- **协同**:风险突变(信任/风险引擎)→ ① 写吊销表 → PoP 切流(权威);② 实时通道 → Agent 强制重认证/重采姿态(提速 + 拿新信号重评)。二者并发,互补不替代。

依据:零信任"判定/执行在受控侧、端不可信";实时通道补"端侧响应速度",但不动权威点,避免把安全寄托于可被攻陷的端。

**风险** 误以为实时通道=撤销手段 → 文档/评审强调权威在 PoP、通道仅提速;通道与吊销表重复下发 → 各自语义(切流 vs 端指令)不冲突、并发。

**结论** 三层分工:吊销表(PoP 权威切流)+ 实时通道(端提速/UX,非权威)+ 短 TTL(不可达兜底);权威永远在 PoP,实时通道不改变安全边界。

### 3.5 多租户与安全边界

**背景** SPA 与实时通道都跨租户,需隔离;且都涉及"端"这一不可信侧。

**目标** 定二者的多租户隔离与安全定位。

**设计**
- **SPA 多租户**:敲门携设备身份 → 解出 `tenant_id` → 授权源进对应租户域(PoP L2 3.3);跨租户不可能(身份加密绑定,L1 3.2)。
- **实时通道多租户**:按设备/租户身份鉴权;指令只对**本租户**设备;心跳/存活按租户命名空间。
- **安全定位(贯穿)**:
  - SPA 是**网络层准入硬化**(隐身/抗扫描),**不替代**隧道认证/PEP——它只是"先认证才让你够到 PoP"。
  - 实时通道是**提速/UX**,**权威在 PoP**(3.4);端不可信。
- **国密**:SPA 敲门加密、实时通道 mTLS 走国密(算法敏捷,待 PoC-G)。实时通道是 **Agent↔控制面**,属国密选型 3.8 的**控制面内部/控制信令面**国密 mTLS(若经 PoP 中继则相应面,见 AH2);SPA 敲门验签算法同端侧栈。

依据:二者都按租户身份隔离(与全平台一致);明确"硬化/提速,非替代权威",避免安全错觉。

**风险** SPA 敲门身份伪造 → 验签(设备证书,不可伪造)+ nonce/时间戳;实时通道被冒充 → mTLS + 凭证鉴权。

**结论** SPA/实时通道均按租户身份隔离;SPA=网络准入硬化(非替代认证)、实时通道=提速(非替代 PoP 权威);加密走国密敏捷。

## 四、风险

### RH1:SPA 敲门洪水/放大攻击
缓解:XDP 廉价预过滤 + 限速先扛,crypto 验签在用户态且限速(3.2);默认丢包使扫描无果。

### RH2:把实时通道当撤销/安全边界
缓解:3.4 明确权威在 PoP、通道仅提速;通道断/端不配合时吊销表+短 TTL 仍强制。

### RH3:SPA 加密/实时通道加密待国密阻塞
缓解:仅加密算法待 PoC-G(算法敏捷、同端侧/数据面栈);SPA 流程/XDP 落地、通道机制不阻塞、可先设计。

### RH4:实时通道海量持久连接规模
缓解:水平扩展 + 心跳(类 xDS server);存活 Redis;离线转待发。

### RH5:SPA 首连延迟 / 白名单规模
缓解:首连多一次往返(小,非每包,L1 3.19);白名单 TTL 自动回收 + 上限。

### RH6:落到端的指令被攻陷端忽略
缓解:端不可信是前提;权威切流在 PoP(3.4),端忽略指令不影响 PoP 强制。

## 五、结论与落点

**结论:** 基于通用零信任原则补两项接入硬化——**① SPA 单包授权**:PoP 接入端口默认丢包(隐身),Agent/CPE 先发加密签名单包敲门(设备身份+租户+时间戳+nonce),**XDP 默认丢包 + 用户态验签 + 短 TTL 白名单**(贴我们 eBPF 栈、不用 iptables),缩小攻击面/抗扫描 DDoS;**② 终端实时控制通道**:Agent 持久 mTLS 通道 + 服务端推送,秒级 撤销/重认证/按需重采姿态/配置/重选路,**权威仍在 PoP PEP、通道仅提速、短 TTL 兜底**(三层分工不重复)。二者按租户身份隔离、加密走国密敏捷(待 PoC-G)、不做 EDR/DLP 本体。

**落点(b 回写各 L2):**

| 落点 | 内容 |
|------|------|
| **PoP 单机编排 L2**(扩展) | SPA:XDP 默认丢包 + 用户态验签 + 白名单 map(3.1/3.2);实时通道在 PoP 的关系(权威切流不变,3.4) |
