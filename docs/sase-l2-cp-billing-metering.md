## 一、背景

L1 3.17 定了计费方向:PoP 采集每租户用量 → 遥测聚合 → `UsageRecord`(L1 3.3);计费**按席位为主 + 带宽超量**(C9);计量须**准确且防篡改**(收入相关)→ 多点核对 + 签名;出账对接成熟计费系统。总览把这些归入 `billing` 模块、点名计量聚合。评审已定 C8(活跃=月内有会话)、C9(席位+带宽超量)。

但"活跃口径怎么实现(跨 PoP/多设备去重)、带宽怎么量、计量在 PoP 可能失陷下怎么防篡改、本模块算到哪一步(raw 用量 vs 出账)"尚未定义。本文回答这些——核心是**在不可全信的计量源(PoP)上得到可用于收入的准确用量**。

---

## 二、目标(可衡量)

1. **计量对象与采集点**:枚举计量 metric、各自采集点与流向。
2. **活跃口径(C8)**:实现"自然月内有会话即活跃",且**同一用户跨 PoP/多设备去重计一席**。
3. **带宽计量(C9)**:per-tenant 字节准确累计;区分**套餐带宽包 vs 超量**;区分**配额限速 vs 超量计费**。
4. **准确性(收入命门)**:计量**防篡改**(PoP 上报签名)+ **多点核对**(两路独立来源比对)+ **幂等**(不重复计/不漏计);偏差可对账。
5. **UsageRecord**:定模型、聚合窗口、落库(经 data 层 InTx、受 RLS)。
6. **定价边界**:本模块产**准确用量(可到 rated 金额)**,**出账/发票/对账对接外部**(L1 3.17)。
7. **多租户隔离 + 配额联动**:UsageRecord 按租户 RLS;计量数据喂单位经济性监控(L1 5.1)。

**非目标:** 出账/发票/收款(外部);价格表数值(商务);数据面限速实现(配额,PoP L2);PoP 计数器实现(PoP L2);单位经济性商业决策(L1 5.1)。

## 三、设计

### 3.1 计量对象与采集点

**背景** 先定清"量什么、在哪量",计费才有据。

**目标** 枚举 metric、采集点、流向,区分"数据面量"与"控制面量"。

**设计 —— 计量对象**

| metric | 含义 | 采集点 | 备注 |
|--------|------|--------|------|
| `bytes_in`/`bytes_out` | 每租户带宽用量 | **PoP**(每租户路由域计数,L1 3.2) | 超量计费基础(3.3) |
| `seats` | 月内活跃用户数(去重基数) | **控制面 `identity`**(凭证签发=会话)+ PoP 会话事件核对 | 席位计费基础(3.2);`seats` 对齐 L1 3.3 `metric` 示例 |
| `connectors`/`cpes` | 连接器/CPE 数 | 控制面 `resource` 模块 | 可选计费项 |
| `feature_usage` | 安全能力用量(SWG/DLP 等) | PoP(可选,按能力) | 后续计费项,起步可不计费 |

- **数据面量(bytes)在 PoP 采集**(它在流量路径上);**身份量(seats)在控制面权威**(它是会话起点,3.2);二者来源不同,核对方式也不同(3.5)。
- 流向:PoP 计量 → 遥测管道(单元③)→ `billing` 聚合 → `UsageRecord`(3.4)。

依据:计量对象与 L1 3.17/3.3 一致;采集点按"谁在权威位置"分(带宽在 PoP、席位在控制面),为 3.2/3.5 的实现与核对奠基。

**风险** 能力用量计量复杂、起步价值低 → `feature_usage` 起步只采集不计费(标注),计费项后续启用;metric 扩展 → `UsageRecord.metric` 可扩(L1 3.3)。

**结论** 计量四类(带宽/席位/连接器 CPE/能力);带宽在 PoP、席位在控制面权威;经遥测管道汇入 billing;能力用量起步只采集。

### 3.2 活跃用户口径实现(C8)

**背景** C9 以席位为主,席位=活跃用户;C8 已定"自然月内有会话即计费"。难点是**同一用户多设备、跨多个 PoP**时不能重复计席。

**目标** 实现月活跃去重计数,定权威信号源。

**设计**
- **活跃信号 = 控制面 `identity` 签发会话凭证**(身份子 L2:IdP 认证→令牌交换→签发短 TTL 凭证,即会话开始)。`identity` 在权威位置知道"哪个 `user_id` 本月产生过会话"。
- **去重计席:** 按 **`user_id` 全局去重**——同一用户多设备、跨多 PoP 的多个会话,**本月计一席**(落实 L1 3.17 风险缓解"按 `user_id` 全局去重计一席");`UsageRecord`/活跃集合的落库经数据访问层子 L2 `InTx`、受 RLS。
- **实现:** `billing` 维护每租户每月的**活跃 `user_id` 集合**(由 `identity` 的凭证签发事件驱动写入,幂等:同 user 当月重复签发不增计);月末/出账时取集合基数 = `active_seats`。
- **PoP 会话事件作核对(非权威):** PoP 的会话建立事件用于**核对**(3.5),不作席位权威来源——因跨 PoP 去重在 PoP 侧做不到(无全局视图)。

依据:凭证签发是会话起点且在控制面(全局视图),按 `user_id` 去重天然解决多设备/跨 PoP 重复;PoP 无全局视图,只能核对不能权威。
- 备选:PoP 侧会话计数汇总。落选:跨 PoP 同一用户无法去重(每 PoP 各算一次),高估席位、计费纠纷。
- 备选:按"注册用户数"计费。落选:违背 C8(已定按活跃),且让租户为不活跃账号付费,不合理。

**风险** 凭证刷新(身份子 L2 3.6 静默刷新)被误计为新会话 → 去重按 `user_id`+月份,刷新不增计;`identity` 事件丢失致漏计 → 与 PoP 会话事件核对(3.5)补偵、月末对账。

**结论** 活跃席位 = 控制面 `identity` 凭证签发驱动、按 `user_id`+月去重的集合基数;PoP 会话事件仅核对;刷新不重复计。

### 3.3 带宽计量与超量(C9)

**背景** C9 含"带宽超量";L1 5.1 指带宽主导成本。需定带宽怎么量、超量怎么界定,并厘清与配额限速的区别。

**目标** 定 per-tenant 带宽计量与超量计费基础,区分配额限速与超量计费。

**设计**
- **计量:** 每租户 `bytes_in/out` 在 **PoP 按租户路由域计数**(L1 3.2 隔离边界天然是计量边界);一个租户跨多 PoP 的字节在 `billing` 累加(per-tenant 全局)。
- **套餐带宽包 vs 超量:** 租户套餐档由 `Tenant.plan`(L1 3.3 标量字段)标识,**套餐包定义(含带宽包 GB/月或 Mbps)在价格表配置**(非 L1 已有数据模型,schema 待定 LB8);**实际用量超出包 = 超量**,按 C11 单价(待报价)计超量费。
- **配额限速 ≠ 超量计费(关键区分):**

| | 配额限速 | 超量计费 |
|--|---------|---------|
| 是什么 | 速率上限(`Quota.max_bandwidth_mbps`,L1 3.3 模型 / 3.2 强制) | 用量超套餐包的计费(C9) |
| 在哪 | **数据面 PoP** 用 tc/eBPF 限速(L1 3.2) | **控制面 billing** 按量计费 |
| 目的 | 防吵闹邻居(QoS) | 收入 |
| 本文 | 只引用(实现属 PoP L2) | 本模块负责 |

依据:带宽计量边界复用 L1 3.2 租户路由域(隔离=计量边界);限速(QoS,数据面)与超量(计费,控制面)是两件事,混淆会导致"限了速还按超量收"或"计费触发限速"的误设。
- 备选:在控制面按流日志估带宽。落选:流日志为采样/聚合,字节数不如 PoP 计数器准(计费需准);流日志作核对(3.5)。

**风险** 带宽计量与限速口径不一致 → 二者都基于同一 per-租户路由域字节,口径同源;跨 PoP 累加时钟窗口错位 → 按统一聚合窗口(3.4)对齐。

**结论** 带宽 per-tenant 在 PoP 计数、billing 跨 PoP 累加;超量=用量超套餐包(按 C11 单价);配额限速(数据面 QoS)与超量计费(控制面收入)严格区分。

### 3.4 计量管道与 UsageRecord

**背景** 总览 3.1:遥测管道→billing 回流计量数据。需定管道与产物模型。

**目标** 定计量数据流、聚合窗口、`UsageRecord` 模型与落库。

**设计**
- **管道:** PoP 计量(带宽计数器、会话事件)→ **遥测管道(单元③)** 摄取(L1 3.1 PoP 上报遥测的一类)→ `billing` 按租户/窗口**聚合** → 产 `UsageRecord`(L1 3.3)→ 经 **data 层 `InTx` 落库**(数据访问层子 L2,受 RLS)。
- **UsageRecord 模型(L1 3.3):** `id, tenant_id, metric(bytes/seats/...), value, window_start, window_end`。带宽按窗口累加值;席位按月去重基数(3.2);窗口对齐(如小时/日聚合 + 月汇总)。
- **聚合窗口:** 带宽按周期(如小时)累加、月汇总;席位按月(C8 自然月);窗口边界统一(UTC+8 自然月,**待确认时区/账期**)。
- **落库:** `UsageRecord` 是租户作用域(含 `tenant_id`),经 `InTx` 受 RLS(数据访问层子 L2);写入幂等(3.5)。

依据:复用遥测管道(单元③)摄取避免另造计量上报通道;UsageRecord 沿用 L1 3.3 模型;窗口聚合支撑账期与对账。

**风险** 遥测管道采样致计量失真 → **计量数据不采样**(与可观测指标区分,计量需全量准确,3.5);聚合窗口与账期不一致 → 账期边界明确(附录)。

**结论** PoP 计量→遥测管道→billing 聚合→UsageRecord(经 data 层 InTx 落库,受 RLS);窗口聚合支撑账期;计量数据全量不采样。

### 3.5 准确性:防篡改 + 多点核对 + 幂等

**背景** L1 3.17 要求计量"准确且防篡改";计量源在 PoP,而 PoP 可能失陷(L1 3.12)。这是收入命门。

**目标** 定防篡改、多点核对、幂等三机制,使收入用量可信、可对账。

**设计 —— 三机制**
1. **防篡改(签名):** PoP 上报的计量数据由 **PoP 用其证书(PoP CA,L1 3.5)签名**;`billing` 验签,拒绝未签/验签失败的上报 → 防途中篡改与伪造源。
2. **多点核对(两路独立来源比对):**
   - 带宽:**PoP 路由域计数器(权威计费源)** vs **遥测管道从流日志聚合的字节数(独立路径)** → 比对;偏差超阈值**告警 + 转人工对账**,不静默采信单一来源。
   - 席位:**控制面 `identity` 凭证签发集合(权威)** vs **PoP 会话事件**(3.2)→ 比对补偵漏计。
   - 依据:单一来源(PoP 自报)若 PoP 失陷可造假;两路独立来源比对使造假/错误需同时突破两路才不被发现(对接 L1 3.12 限爆炸半径)。
3. **幂等(不重复/不漏):** 计量上报带**窗口 id + 序号**;`billing` 按 (tenant, metric, window) **幂等合并**——PoP 重传不重复计;缺窗口检测漏报并补拉/告警。

- **对账:** 偏差与缺口产对账记录,供人工/外部计费系统核对(L1 3.17 出账对接外部)。

依据:签名防伪造、多点核对防单源造假、幂等防重复/漏计——三者共同满足 L1 3.17"准确且防篡改"且适配 PoP 不可全信(L1 3.12)。
- 备选:只信 PoP 单一上报。落选:PoP 失陷即收入数据被操纵,无核对手段,违背 L1 3.17/3.12。

**风险** 两路长期小偏差(计数口径差异)→ 设容差阈值 + 定期校准,超阈才告警;签名密钥(PoP 证书)管理 → 复用 L1 3.5 PoP 证书 + 撤销;遥测管道若采样则核对失真 → 计量路径不采样(3.4)。

**结论** PoP 签名上报 + 带宽/席位各两路独立核对 + 窗口幂等合并 + 偏差对账;在 PoP 不可全信前提下得到可用于收入的准确用量。

### 3.6 计费模型与定价边界(C9)

**背景** C9 已定"按席位为主 + 带宽超量"。需定本模块算到哪一步、什么交外部。

**目标** 定计费模型落地与本模块/外部计费系统的边界。

**设计**
- **计费模型(C9):** 月费 = 席位单价 × `seats`(3.2 月活跃去重基数) + 带宽超量费(3.3,超套餐包部分 × C11 单价) + (可选)连接器/CPE 项。席位为主(L1 5.1:席位价须覆盖带宽成本)。
- **本模块边界(到 rated usage 为止):** `billing` 产 **准确用量 `UsageRecord`(raw)** + 套用价格表算出的 **rated 金额(计费就绪数据)**;**出账、发票、收款、对账、税务对接外部成熟计费系统**(L1 3.17)。
- **价格表:** 价格表(席位单价、带宽超量单价、套餐定义)为配置数据(商务定,数值待确认/C11 待报价);本模块消费价格表算 rated,不定价格数值。

依据:L1 3.17 明确"本设计只保证计量数据准确可用,出账对接成熟计费系统"——故本模块到 rated usage 为止,避免重造出账/发票/税务(成熟系统更擅长)。
- 备选:本模块自建出账/发票。落选:出账/税务/对账是成熟商业软件领域,自建成本高、合规(发票)复杂,L1 已定对接外部。

**风险** 价格表变更影响在途账期 → 价格表版本化 + 账期锁价;rated 与外部系统口径不一致 → 以 raw UsageRecord 为对账基准,rated 仅为就绪数据。

**结论** 计费=席位×单价 + 带宽超量(+可选项);本模块产 raw UsageRecord + rated 金额,出账/发票/对账交外部;价格表为配置、数值商务定(C11 待报价)。

### 3.7 多租户隔离与配额联动

**背景** UsageRecord 是租户数据;配额(L1 3.2)与计量相关但目的不同;计量数据还喂单位经济性(L1 5.1)。

**目标** 定计量的多租户隔离、与配额的联动、对单位经济性的供数。

**设计**
- **隔离:** `UsageRecord`/活跃集合/对账记录均含 `tenant_id`、受 RLS(数据访问层子 L2);租户管理员只见本租户用量(L1 3.18 控制台)。
- **配额联动(区分但相关):** 配额(`Quota`:带宽/连接器/策略数上限,L1 3.2)由 PoP/控制面**强制**(限速/拒绝);计量是**事后用量**。二者数据相关(都看带宽)但动作不同:配额=实时管控,计量=计费。超配额→限速(数据面);超套餐包→超量计费(本模块)。
- **供数单位经济性(L1 5.1):** 聚合用量(带宽/席位/人均吞吐)供**单位经济性监控**(每用户带宽成本、自建拐点输入 C10)——本模块**供数据**,**商业判断属 L1 5.1**(非本模块决策)。

依据:计量数据按租户隔离与全平台一致(RLS);配额与计费虽都涉带宽但动作分明(管控 vs 计费);单位经济性是商业判断、本模块只供数。

**风险** 租户管理员越权看他租户用量 → RLS + authz(数据访问层子 L2 / L1 3.18);计量供数被误当商业结论 → 标注"供数据,判断见 L1 5.1"。

**结论** UsageRecord 受 RLS 隔离;配额(实时管控)与计费(事后计量)区分;计量聚合供单位经济性监控,商业判断属 L1 5.1。

---

## 四、风险

### RB1:PoP 失陷致计量被操纵(收入命门)
缓解:PoP 签名上报 + 带宽/席位两路独立核对 + 偏差告警对账(3.5);PoP 证书可撤销(L1 3.5/3.12)。

### RB2:跨 PoP/多设备重复计席
缓解:席位按 `user_id` 全局去重(落实 L1 3.17)、控制面 identity 权威(3.2);PoP 不作席位权威源。

### RB3:计量不准致收入漏损/纠纷(L1 3.17)
缓解:计量全量不采样(3.4)+ 多点核对 + 幂等(3.5)+ 以 raw UsageRecord 为对账基准(3.6)。

### RB4:配额限速与超量计费混淆
缓解:3.3/3.7 明确区分——限速在数据面(QoS)、超量在控制面(计费),口径同源但动作分离。

### RB5:遥测管道采样使核对失真
缓解:计量路径与可观测指标分离,**计量不采样**(3.4);核对用全量流日志聚合。

### RB6:账期/时区边界不一致
缓解:统一账期(自然月、时区待确认,附录)+ 窗口对齐(3.4);价格表账期锁价(3.6)。

## 五、结论与衔接

**结论:** `billing` 模块在**计量源(PoP)不可全信**的前提下产出可用于收入的准确用量:计量四类(带宽/席位/连接器 CPE/能力),**带宽在 PoP 计数、席位由控制面 `identity` 凭证签发驱动按 `user_id` 全局去重**(C8);经遥测管道聚合产 `UsageRecord`(经 data 层 InTx、受 RLS);准确性靠 **PoP 签名上报 + 带宽/席位两路独立核对 + 窗口幂等合并 + 偏差对账**(满足 L1 3.17、适配 L1 3.12);计费**按席位 + 带宽超量**(C9),本模块产 **raw UsageRecord + rated 金额**,**出账/发票/对账交外部计费系统**;严格区分**配额限速(数据面 QoS)与超量计费(控制面收入)**;计量供单位经济性监控,商业判断属 L1 5.1。

**衔接:**
- **遥测管道(单元③):** 摄取 PoP 计量,billing 聚合其数据;计量路径不采样(与可观测指标区分)。
- **身份与 IdP 子 L2:** 凭证签发事件 = 活跃席位信号(去重权威源)。
- **数据访问层子 L2:** UsageRecord/活跃集合/对账经 `InTx` 落库、受 RLS。
- **PoP 单机编排 L2:** 带宽计数器、计量上报签名、会话事件——PoP 侧实现,守本文计量契约(签名、窗口、metric)。
- **外部计费系统:** 出账/发票/收款/对账(L1 3.17),以 raw UsageRecord 为基准。
- **L1 5.1:** 供单位经济性数据(每用户带宽成本、拐点输入 C10)。
- 与国密无关(不依赖 PoC-G)。
