## 一、背景

L1 3.9 定 SD-WAN 为**云交付**:不卖重 CPE,优先**软件 CPE**(跑客户现有硬件/虚机的 Go agent),选路价值在 PoP 骨干;CPE 建到 PoP 的叠加隧道、多 WAN 探测、应用感知选路 + 劣化亚秒切换、ZTP;链路修复(FEC/包复制)仅对实时业务 + 劣化链路开启(否则放大带宽成本 R3)。SD-WAN 复用控制面/策略/xDS/遥测,排 ZTNA 之后(P3)。

但 CPE 内部怎么组织(WAN 怎么探、选路 CPE 与 PoP 怎么分工、ZTP 怎么走、加密栈怎么处理国密、配置怎么下发)尚未定义。本文给 CPE L2。核心约束:**软件 CPE、选路重心在 PoP 骨干、CPE 本地只做需快速反应的部分、加密栈待国密但算法敏捷**。

---

## 二、目标(可衡量)

1. **软件 CPE 形态**:跑客户现有硬件/虚机的 Go agent,避免重硬件包袱(L1 3.9)。
2. **多 WAN 探测**:持续测 RTT/抖动/丢包(`CPE.wan_links`,L1 3.3),供选路与切换。
3. **选路分工**:CPE 本地做需亚秒反应的链路切换;跨站点/出口的全局选路在 PoP 骨干(L1 3.9)。
4. **劣化亚秒切换**(L1 3.22 P3 验收)。
5. **链路修复**:FEC/包复制/抖动缓冲,**仅实时业务 + 劣化链路**(控带宽,R3)。
6. **ZTP < 30min**(C3/L1 3.22):上电→回连→激活码+CSR→tenant-bound 证书→拉配置;**私钥本地生成永不离开**(L1 3.5)。
7. **加密栈算法敏捷**:隧道加密随国密选型(crypto provider),**待 PoC-G**,不锁算法。
8. **契约清晰**:与控制面(配置/证书)、PoP(隧道/选路)、遥测的契约。

**非目标:** PoP 骨干选路实现(PoP L2);加密算法选定(国密 PoC-G);写代码;硬件适配清单(实施期);重 CPE。

## 三、设计

### 3.1 CPE 形态与部署

**背景** L1 3.9 已定优先软件 CPE。需明确部署形态与对客户环境的要求。

**目标** 定 CPE 部署形态,使其能跑在客户现有环境、避免硬件包袱。

**设计**
- **软件 CPE = Go agent**,部署形态:跑在客户现有 x86 软路由/服务器、虚机、或容器;一个站点一个 CPE 实例(可主备)。
- **对客户环境要求最小**:出站可达 PoP(隧道,3.5)、管理多个 WAN 口(3.2);不要求公网入站(出站建隧道,同 ZTNA App Connector 哲学)。
- **不卖重硬件**:依据 L1 3.9——重 CPE 有供应链/库存/RMA 包袱;软件 CPE 复用客户硬件、快速铺开。
  - 备选:自产硬件 CPE。落选(起步):供应链/库存/RMA 成本,与"云交付、轻接入"定位冲突(L1 3.9);若客户要求一体机,后续可出硬件形态(同一 Go agent 预装)。

**风险** 客户现场硬件/网络长尾兼容 → 软件 CPE 优先 + 兼容性测试矩阵(实施期);CPE 单点 → 站点可双 CPE 主备(待确认 HA 形态)。

**结论** 软件 CPE(Go agent,跑客户现有硬件/虚机/容器),出站建隧道、最小环境要求;不卖重硬件,HA 主备待确认。

---

### 3.2 WAN 链路管理与探测

**背景** SD-WAN 价值前提是"看清每条 WAN 的实时质量"。L1 3.3 `CPE.wan_links(jsonb)`、3.9 多 WAN 探测。

**目标** 定多 WAN 的管理与持续探测,产选路/切换所需的链路质量数据。

**设计**
- **多 WAN 管理:** CPE 管理多条 WAN(宽带/4G-5G/MPLS),每条记录于 `wan_links`(L1 3.3),含类型、状态、本地计量参考。**站点带宽计量归数据面 PoP + 计费计量子 L2(C8/C9),CPE 侧仅本地参考、不作计费源。**
- **持续探测:** 对每条 WAN 持续测 **RTT / 抖动 / 丢包**,用**主动探测包 + 被动流量统计混合**;频率/窗口可配(LS4)。依据:纯主动探测开销大(尤其蜂窝计费链路)、纯被动在空闲链路无数据且劣化反应慢 → 混合兼顾及时性与开销。备选纯主动/纯被动均落选(前者贵、后者迟钝)。
- **质量评分:** 由探测指标算每条 WAN 的质量分,供本地切换(3.3)与上报选路数据(3.7);评分算法可配。

**风险** 探测自身耗带宽(尤其 4G/5G 计费链路)→ 探测包小、频率自适应(劣化时加密、稳定时降频);蜂窝链路计费 → 探测计入成本、可对计费 WAN 降探测频率。

**结论** CPE 管理多 WAN(`wan_links`)、持续探测 RTT/抖动/丢包并评分;探测频率自适应控开销(蜂窝链路尤甚)。

### 3.3 选路:CPE 本地 vs PoP 骨干分工

**背景** L1 3.9 明示"选路价值在 PoP 骨干";但劣化切换需亚秒反应(L1 3.22),不能等控制面/PoP。分工是关键。

**目标** 定 CPE 本地与 PoP 骨干的选路职责边界。

**设计 —— 两层分工**
- **CPE 本地(需快速反应的部分):**
  - **WAN 上行选择 + 劣化亚秒切换**:本地按实时质量分(3.2)选哪条 WAN 上 PoP、劣化时**亚秒切换**到次优 WAN(本地决策,不等远端,满足 L1 3.22)。
  - **应用感知分流(本地策略)**:按下发的应用感知选路策略(来自控制面,3.7),把不同应用导向不同路径(如实时业务走低抖动 WAN)。
- **PoP 骨干(全局视图的部分):**
  - **跨站点/出口选路**:站点 A→站点 B 走哪条 PoP 间骨干、或站点→SaaS/互联网经哪个 PoP 出口——**全局视图,在 PoP 骨干**(L1 3.9),**属 PoP L2**。
- **依据:** 亚秒切换必须本地(远端往返来不及);跨站点/出口选路需全局拓扑视图(单 CPE 没有),归 PoP 骨干。二者分工 = "本地快反应 + 骨干全局优"。
  - 备选:全在 CPE 本地选路。落选:单 CPE 无全局拓扑,跨站点选路不优(L1 3.9 价值在骨干)。
  - 备选:全在 PoP/控制面选路。落选:劣化切换等远端往返,达不到亚秒(L1 3.22)。

**风险** 本地与骨干选路决策冲突 → 本地只管"上哪条 WAN 到 PoP",骨干管"PoP 间/出口",层次不重叠;应用识别不准致分流错 → 应用识别策略可配 + 默认路径兜底。

**结论** CPE 本地做 WAN 选择 + 劣化亚秒切换 + 应用感知分流(按下发策略);跨站点/出口全局选路在 PoP 骨干(属 PoP L2);层次不重叠。

### 3.4 链路修复(FEC / 包复制 / 抖动缓冲)

**背景** L1 3.9:FEC、包复制(双发取先到)、抖动缓冲在丢包链路保语音/视频;**仅对实时业务 + 劣化链路开启**(否则放大带宽成本 R3)。

**目标** 定链路修复机制与开启条件,在保质量与控带宽间平衡。

**设计**
- **机制:** FEC(前向纠错,冗余包补丢包)、包复制(同包多 WAN 双发、取先到)、抖动缓冲(平滑到达抖动)。
- **开启条件(关键,控带宽):** **仅当 ①业务为实时类(语音/视频,按应用识别)且 ②链路劣化(丢包/抖动超阈)** 才开启;稳定链路或非实时业务**不开**——因 FEC/包复制成倍耗带宽,而带宽主导成本(R3/L1 5.1)。
- **自适应:** FEC 冗余度、是否包复制随劣化程度调;恢复后关闭。

依据:链路修复是"以带宽换质量",只在"值得换"(实时业务遇劣化)时换,否则放大 R3 成本;L1 3.9 已定此约束,本文给开启条件。
- 备选:全程开 FEC/包复制。落选:带宽成本成倍、与 R3/单位经济性冲突。

**风险** 实时业务识别不准致该开没开/不该开却开 → 识别 + 可配阈值 + 保守默认(拿不准不开,省带宽);包复制需多 WAN 同时可用 → 单 WAN 时退化为 FEC。

**结论** FEC/包复制/抖动缓冲**仅实时业务 + 劣化链路**开启、自适应、恢复即关;以最小带宽代价保实时质量,守 R3。

### 3.5 隧道与加密栈(形态已定,加密算法待国密)

> **隧道形态修正(随 L1 v0.6 / gm-crypto v0.4 / `sase-l2-data-plane-tunnel.md` v0.2):** 本节原写「CPE↔PoP 叠加隧道 = WireGuard / TLCP 流隧道双栈」。**SD-WAN 数据面隧道形态已收敛为「自研 crypto-agile UDP 数据报隧道」**(`internal/dptunnel`):AEAD 数据报(`SM4-GCM`↔`ChaCha20-Poly1305` 可插拔,对应国密/非国密双档)+ 计数器重放窗口 + 块状 FEC,**握手用 TLS1.3/TLCP 导出密钥(零自研密钥协商,待密码学审查)**,而非 WireGuard、也非 TLS 记录层流。**国密算法栈(SM4-GCM/SM2 握手/SM3)与 C-G1 硬件要求不变,变的是封装形态**;数据报形态契合 SD-WAN 多链路/FEC/亚秒切换(`linkmon`,3.x)。本节余处 WireGuard / TLCP 流 表述就此修正,**以 `sase-l2-data-plane-tunnel.md` v0.2 为准**。

**背景** CPE 建到 PoP 的叠加隧道(L1 3.9,自研 UDP 数据报隧道,见上修正)。加密栈与客户端数据面同源,受国密影响(R7)。

**目标** 定隧道形态(已定:自研 UDP 数据报),加密算法敏捷、SM4 性能待国密 PoC-G。

**设计**
- **隧道:** CPE↔PoP 叠加隧道(overlay,自研 crypto-agile UDP 数据报),承载站点流量进租户路由域(3.8)。
- **加密栈(算法敏捷 = `dptunnel` AEAD provider 抽象,见 gm-crypto 3.2 / 隧道 L2):** 双档经同一 provider 接口:**国密档租户用 SM4-GCM,非国密租户用 ChaCha20-Poly1305**——与数据面隧道选型(gm-crypto 3.2 v0.4)及客户端数据面 **同一套 AEAD provider 抽象**;握手 TLS1.3/TLCP 导出密钥。**SM4 带加速吞吐待 PoC-G(M-G3~M-G5);握手待密码学审查。**
- **私钥本地生成:** 隧道密钥/证书私钥**在 CPE 本地生成、永不离开**(L1 3.5),只注册公钥(ZTP,3.6)。

依据:CPE 隧道与客户端数据面隧道同源,复用 `dptunnel` provider 抽象,避免 CPE 自成一套;私钥本地生成守 L1 3.5 贯穿原则。

**风险** 国密隧道性能(SM4)影响 CPE 转发 → 随 PoC-G 实测(且 CPE 跑客户硬件、SM4 加速不确定,更需实测);握手密码学审查未过阻塞真密钥接线 → **加密算法/握手这一块待 PoC-G/审查**,CPE 其余(WAN/选路/ZTP/修复)不阻塞、已设计实现。

**结论** CPE↔PoP 叠加隧道 = **自研 crypto-agile UDP 数据报隧道(SM4-GCM↔ChaCha20-Poly1305 双档、握手 TLS1.3/TLCP 导出密钥)**、与数据面同源、私钥本地生成;**加密算法 SM4 性能 + 握手密码学审查待 PoC-G**,其余 CPE 设计不受阻。

### 3.6 ZTP 零接触开通

**背景** L1 3.9/3.5:ZTP(上电→回连→认证 tenant-bound 证书 + 一次性激活码→拉配置);C3/L1 3.22 要求 < 30min。

**目标** 定 ZTP 流程,满足零接触 + <30min + 安全(私钥本地生成、tenant-bound)。

**设计 —— ZTP 流程**
1. **上电回连:** CPE 首次上电,用预置的引导信息(激活码 + 控制面地址)发起回连(出站)。
2. **认证 + 签发:** CPE 本地生成密钥对 + **CSR**(私钥不离设备,L1 3.5);凭**一次性激活码 + CSR** 向控制面注册 → 控制面校验激活码、签发 **tenant-bound CPE 证书**(CPE CA,L1 3.5,绑 `tenant_id`+`site_id`)。**注册协议复用 L1 3.11 Connector `Register{activation_code, csr}` 契约模式**(L1 3.5 已把 Connector/CPE 并列;L1 3.11 已于 **v0.5 把 CPE 与 Connector 并列(W5)**)。
3. **拉配置:** CPE 用证书拉取:租户路由域配置、WAN 配置模板、选路/应用感知策略、隧道目标 PoP 列表(3.7)。
4. **建隧道 + 上线:** 按配置建到 PoP 的隧道(3.5)、开始探测(3.2)与转发。
- **<30min(C3):** 全程零现场配置;激活码可由租户管理员在控制台预生成(前端控制台 L2)。
- **激活码安全:** 一次性、时效、绑站点;用尽即失效(防盗用)。

依据:沿用 L1 3.5/3.9 ZTP;私钥本地生成 + tenant-bound 证书 + 一次性激活码 = 零接触且安全;<30min 靠全程免现场配置。
- 备选:预置长期证书出厂。落选:出厂预置私钥有泄露与盗用风险,违背"私钥本地生成"(L1 3.5)。

**风险** 激活码泄露被盗开通 → 一次性 + 时效 + 绑站点 + 审计(L1 3.18);ZTP 期间网络不通 → 重试 + 引导诊断;首配拉取慢 → 配置精简 + <30min 验收(L1 3.22)。

**结论** ZTP:上电回连→激活码+CSR→tenant-bound 证书(私钥本地生成)→拉配置→建隧道,全程零现场、<30min;激活码一次性时效绑站点。

### 3.7 与控制面/PoP 的配置与遥测契约

**背景** L1 3.9"复用控制面/策略/xDS/遥测"。CPE 作为边缘端,需定它与控制面/PoP 的契约。

**目标** 定 CPE 的配置下发、策略、遥测上报契约,及与控制面 L2 的衔接。

**设计**
- **配置下发:** ZTP 拉取(3.6)+ 运行期增量更新:租户路由域、WAN 配置、**应用感知选路策略**、隧道目标 PoP 列表。传输**复用 xDS 机制承载 CPE 配置自定义资源**(类比 L1 3.1 非 Envoy 组件复用 xDS 传输)或经控制面控制通道下发。**两个待对齐点(附录 LS2/LS10):① CPE 经控制面直连还是经 PoP 中继获取配置;② 若走 xDS 自定义资源,需在 xDS server 子 L2 的资源清单(当前 6 类)登记为新增类型**(类比其 `TrustBundle` 的登记方式),避免两文档对资源类型数打架。
- **选路策略:** 应用感知选路策略由控制面下发(可能复用策略编译器产物的一部分,或独立的选路策略——**待确认是否纳入策略编译器**);CPE 本地执行(3.3)。
- **遥测上报:** CPE 上报 WAN 质量(RTT/抖动/丢包)、隧道状态、切换事件到遥测管道(单元③);**WAN 质量数据供 PoP 骨干选路(PoP L2)与运维**。
- **证书/身份:** tenant-bound CPE 证书(L1 3.5);控制通道 mTLS(国密档走国密 mTLS,对接国密 3.8 内部面)。

依据:CPE 复用控制面下发/遥测(L1 3.9),不自成体系;配置经 xDS 机制或控制通道,与控制面 L2 契约对齐。

**风险** CPE 配置传输路径(直连控制面 vs 经 PoP 中继)未定 → 列待确认,与 xDS server 子 L2 / PoP L2 一并定;选路策略是否进策略编译器 → 待确认(选路策略与访问策略性质不同,可能独立)。

**结论** CPE 配置经 xDS 机制/控制通道下发(路径待与 xDS 子 L2 对齐)、选路策略控制面下发 CPE 本地执行、WAN 质量遥测上报供 PoP 骨干选路;证书 tenant-bound、控制通道 mTLS(国密档国密化)。

---

### 3.8 路由域与站点寻址

**背景** L1 3.2 每租户独立路由域;3.6 SD-WAN 站点子网经 CPE 在租户路由域内互通。

**目标** 定 CPE 侧的路由域归属与站点寻址。

**设计**
- **租户路由域:** CPE 的隧道流量进**该租户独立路由域**(L1 3.2);跨站点流量在租户路由域内经 PoP 骨干转发(L1 3.6/3.9)。
- **站点子网:** 各站点本地子网经 CPE 接入租户 overlay;站点间在租户路由域内互通(L1 3.6)。
- **地址:** 复用 L1 3.6 租户 overlay 地址(租户内不冲突;租户间路由域隔离可复用);站点子网与 overlay 冲突时用 CGNAT/可配(L1 3.6)。

依据:沿用 L1 3.2/3.6,CPE 只是把站点接入既有租户路由域,不新增隔离机制。

**风险** 站点本地子网与其他站点/overlay 冲突 → 地址规划 + CGNAT(L1 3.6);路由域归属错置致跨租户 → 由 tenant-bound 证书 + 控制面配置确定路由域(L1 3.2/3.5)。

**结论** CPE 流量进租户独立路由域、站点子网在域内经 PoP 骨干互通、复用 L1 overlay 寻址;路由域由 tenant-bound 证书 + 配置确定。

---

### 3.9 可观测与运维

**背景** CPE 在客户现场、数量多,需远程可观测与运维。

**目标** 定 CPE 的状态/质量上报与远程运维。

**设计**
- **上报:** CPE 状态(在线/离线)、各 WAN 质量、隧道状态、切换/修复事件 → 遥测管道(3.7);租户管理员经控制台看本租户站点/CPE 状态(前端控制台 L2)。
- **远程运维:** 配置远程下发(3.7)、CPE agent 远程升级(灰度 + 回滚,类比 L1 3.13 发布);故障诊断信息上报。
- **告警:** WAN 全断、CPE 离线、频繁切换等告警(对接 L1 3.14)。

依据:CPE 量多且在客户现场,远程可观测/运维是可运营前提;复用遥测/发布机制(L1 3.13/3.14)。

**风险** CPE 升级致现场中断 → 灰度 + 回滚 + 维护窗口;CPE 离线无法远程修 → 本地最小自恢复 + 现场兜底流程(实施期)。

**结论** CPE 状态/质量/事件上报遥测、远程配置下发与灰度升级、关键告警;复用 L1 遥测/发布机制。

## 四、风险

### RC1:加密栈待国密阻塞 CPE 加密实现
缓解:**仅加密栈待 PoC-G**(3.5,算法敏捷、与 Agent 同源);CPE 其余(WAN/选路/ZTP/修复/契约)不阻塞、可先设计实现。

### RC2:劣化切换达不到亚秒
缓解:切换决策本地化(3.3)、不等远端;探测自适应加密(3.2);亚秒切换纳入 P3 验收(L1 3.22)。

### RC3:FEC/包复制放大带宽成本(R3)
缓解:仅实时业务 + 劣化链路开启、保守默认(3.4)。

### RC4:ZTP 激活码盗用
缓解:一次性 + 时效 + 绑站点 + 审计;私钥本地生成、证书 tenant-bound(3.6)。

### RC5:CPE 配置传输路径未定
缓解:列待确认,与 xDS server 子 L2 / PoP L2 契约一并定(3.7/附录)。

### RC6:客户现场硬件/网络长尾兼容
缓解:软件 CPE 优先 + 兼容矩阵(3.1/LS9)。

### RC7:选路策略是否纳入策略编译器未定
缓解:待确认(3.7);选路策略与访问策略性质不同,可能独立于编译器。

### RC8:CPE 单点(站点失 CPE 即断网)
缓解:站点可双 CPE 主备(HA 形态待确认 LS5);本地最小自恢复 + 现场兜底流程(实施期,3.9)。

---

## 五、结论与衔接

**结论:** SD-WAN CPE = **软件 CPE(Go agent,跑客户现有硬件/虚机/容器)**;多 WAN 持续探测 RTT/抖动/丢包并评分;**选路两层分工——CPE 本地做 WAN 选择 + 劣化亚秒切换 + 应用感知分流,跨站点/出口全局选路在 PoP 骨干(PoP L2)**;链路修复 FEC/包复制/抖动缓冲**仅实时业务 + 劣化链路**开启(控 R3);**ZTP**(上电回连→激活码+CSR→tenant-bound 证书→拉配置,私钥本地生成,<30min);**隧道 = 自研 crypto-agile UDP 数据报隧道,与数据面同源(SM4-GCM↔ChaCha20-Poly1305 双档、握手 TLS1.3/TLCP 导出密钥,见 3.5 修正),仅加密算法 SM4 性能 + 握手密码学审查待国密 PoC-G**;配置/遥测复用控制面(xDS 机制/遥测管道);流量进租户独立路由域(L1 3.2/3.6)。

**衔接:**
- **PoP 单机编排 L2(部分待 PoC-1):** PoP 骨干选路、CPE 隧道在 PoP 侧的终结——属 PoP L2,CPE 守本文隧道/选路契约。
- **国密选型(gm-crypto v0.4 / 隧道 L2 v0.2):** 隧道形态已定(自研 UDP 数据报),加密算法 SM4-GCM↔ChaCha20-Poly1305 双档(算法敏捷),SM4 性能待 PoC-G、握手待密码学审查。
- **xDS server 子 L2:** CPE 配置下发(资源/路径待对齐)。
- **策略编译器子 L2:** 应用感知选路策略是否纳入(待确认)。
- **遥测管道(单元③):** CPE WAN 质量/状态上报(供 PoP 骨干选路与运维)。
- **身份/PKI(L1 3.5):** tenant-bound CPE 证书、控制通道 mTLS(国密档国密化,国密 3.8 内部面)。
- **前端控制台 L2:** 站点/CPE 开通(激活码生成)、状态看板。

搭脚手架 / 写 CPE 代码须**另行授权**(设计先行)。
