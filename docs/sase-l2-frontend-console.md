## 一、背景

L1 3.18 定了**两类控制台**——平台运维控制台(我方:PoP/租户/容量/安全运营)与租户管理员控制台(客户:本租户用户/应用/策略/站点/用量/审计),前端 TS/React,后端 Admin API(L1 3.11);并定**授权在后端强制、前端鉴权不可信、全审计**。

但前端内部怎么组织(双控制台是一套还是两套、模块/路由怎么分、怎么消费 Admin API 不漂移、前端 RBAC 做到哪一步、关键交互如策略编辑怎么呈现)尚未定义。本文给前端 L2 内部架构。核心约束:**前端是 Admin API 的消费者与体验层,不是安全边界**(安全在后端,L1 3.18)。

---

## 二、目标(可衡量)

1. **双控制台边界与复用**:平台/租户两控制台的代码组织,既复用又隔离。
2. **模块/路由结构**:按业务域划分,与控制面模块/数据对齐。
3. **契约消费**:以 OpenAPI(L1 3.11 单一来源)生成前端类型与 client,**契约漂移在 CI 拦截**。
4. **前端 RBAC**:角色驱动 UI 可见性,**所有授权后端强制**(L1 3.18),前端只做体验层收敛。
5. **数据层**:以 server-state(远端数据)为主,client/UI state 最小,定缓存/失效/刷新策略。
6. **关键交互**:策略编辑(求值顺序 + 遮蔽/冲突告警)、用量看板、审计查询、IdP 配置的设计点落到对应控制面子 L2。
7. **横切**:统一错误处理、中文为主 i18n、前端可观测、传输安全(构建形态归技术取向 3.8)。

**非目标:** 写代码/脚手架;视觉稿;后端授权(控制面 L2);库版本/组件清单数值(3.8 待确认);国密 TLS 实现(前置终结)。

## 三、设计

### 3.1 双控制台边界与代码组织

**背景** L1 3.18 两类控制台用户群、权限、部署不同,但有大量共享(设计系统、API 客户端、类型、通用组件)。组织方式决定复用与隔离的平衡。

**目标** 定两控制台的代码组织,既最大化复用又防止平台/租户能力混入对方。

**设计 —— 单前端代码库 + 共享层 + 两应用入口**
- **共享层(复用):** 设计系统/通用组件、由 OpenAPI 生成的**类型与 API client**(3.3)、鉴权/请求/错误/i18n 等基础设施。
- **两应用入口(隔离):** `app-admin`(平台运维)与 `app-tenant`(租户管理员)各自独立入口与路由树、独立构建产物、可独立部署到不同域名/网关。
- **平台能力不进租户包:** 平台运维专属模块(PoP/容量/安全运营/跨租户)只在 `app-admin`,**不打进** `app-tenant` 产物——减少租户侧包体与"前端里藏着平台能力"的暴露面。

依据:共享层避免重复(组件/类型/client),双入口隔离用户群与构建产物,平台能力物理上不进租户包(纵深,虽安全靠后端但减少暴露面)。
- 备选:**两个独立 repo**。落选:共享组件/类型/client 重复或跨 repo 依赖,维护成本高。
- 备选:**单一应用按角色切换**(平台/租户同一入口)。落选:平台能力代码进租户包(暴露面 + 包体),且路由/菜单按角色动态裁剪易遗漏,越权 UI 风险高。

**风险** 共享层耦合致一改全动 → 共享层只放稳定的通用件,业务组件留各 app;两入口重复样板 → 脚手架/模板复用(实现期)。

**结论** 单代码库 + 共享层(设计系统/生成类型/client/基础设施)+ `app-admin`/`app-tenant` 两入口;平台能力不进租户包。

### 3.2 模块与路由结构

**背景** 前端模块应与控制面业务域对齐,便于映射 API 与权限。

**目标** 给两控制台的模块/路由划分(设计层,非组件代码)。

**设计 —— 按业务域(与控制面模块对齐)**

| 控制台 | 模块/路由域 | 对应控制面 |
|--------|-----------|-----------|
| 租户(`app-tenant`) | 用户/组、应用(ZTNA)、策略、站点/CPE(SD-WAN)、用量看板、审计日志、IdP 配置、本租户设置 | identity/resource/policy/billing/audit/tenant |
| 平台(`app-admin`) | PoP/容量、租户管理、安全运营、跨租户支持、平台 RBAC/CA 管理(敏感双人控制) | platform/tenant/PKI/admin |

- 路由树按域分层;每域内列表/详情/编辑常规化。
- 租户控制台所有数据**天然限本租户**(后端 RLS + API authz,L1 3.2/3.18),前端不需也不应自己过滤跨租户。

依据:模块按业务域 = 与控制面模块/数据/权限一一映射,降低前后端心智差;租户隔离由后端保证,前端只渲染 API 返回。

**风险** 域划分与控制面模块漂移 → 以 OpenAPI 分组/标签为锚(3.3);平台侧跨租户视图误入租户控制台 → 由 3.1 双入口物理隔离。

**结论** 两控制台按业务域分模块/路由、与控制面模块对齐;租户侧数据由后端限定,前端不自行跨租户过滤。

### 3.3 与 Admin API 的契约消费

**背景** L1 3.11 定 Admin REST + OpenAPI 为**单一来源**、前端为消费者、契约与实现不漂移。

**目标** 定前端如何消费 API、如何防漂移。

**设计**
- **生成而非手写:** 由控制面 `api/openapi`(总览 3.6 单一来源)**生成前端 TS 类型 + API client**;前端业务代码只用生成物,不手写请求类型。
- **防漂移:** OpenAPI 变更 → 重新生成 → 类型不兼容处编译报错(及早暴露);CI 校验生成物与 OpenAPI 一致(对接总览 3.6 "生成 + CI 校验")。
- **错误契约:** 统一错误响应结构(后端定),前端统一拦截处理(3.7)。
- **审计透明:** 写操作经 API 记审计(L1 3.3/3.18),前端无需额外处理,只呈现结果。

依据:生成式消费 + 编译期类型校验是契约漂移的结构性防护(L1 3.11 风险的前端落点);单一来源避免前后端各写一套类型。
- 备选:手写 TS 类型/请求。落选:与后端契约漂移、双份维护,正是 L1 3.11 要避免的。

**风险** OpenAPI 描述不全(如鉴权头、分页约定)致生成物不可用 → OpenAPI 须含完整约定(与控制面 `admin` 模块协同);生成工具选型 → 待确认(3.8)。

**结论** OpenAPI 生成 TS 类型 + client、前端只用生成物、CI 防漂移;不手写请求类型。

> **✅ as-built(Slice33,后端侧):** 选定**手写 spec + CI 一致性测试**(方式①;非 swaggo 从代码生成、非 oapi-codegen——忠于"spec 单一来源"+ 改动最小)。`api/openapi/admin.yaml`(OpenAPI 3.1 手写权威,27 路径)+ `internal/admin/httpapi` 的 `AdminRoutePatterns`(导出权威路由清单,启动 fail-loud assert 与 handler 一致)+ `openapi_conformance_test.go`(spec↔清单双向对账,漂移即 CI 红)。**前端从该 spec 用 openapi-typescript 类生成 TS 类型/client**(前端编码时)。局限:当前仅路径级对账,响应 schema 完整性随端点增长补。

### 3.4 前端 RBAC 与租户隔离(体验层)

**背景** L1 3.18 明确:授权后端强制、前端鉴权不可信。前端 RBAC 只能是体验层。

**目标** 定前端角色驱动的 UI 行为,且不把它当安全边界。

**设计**
- **来源:** 登录后会话携带的角色/权限(平台角色:超管/运维/只读支持/安全/CA 管理员;租户角色:租户管理员/策略管理员/审计员/helpdesk,L1 3.18)用于**菜单/操作/路由的可见与可用**。
- **仅体验层:** 前端隐藏/禁用无权操作只为体验;**真正授权一律后端强制**——前端即便被绕过(改请求、直调 API),后端 authz + RLS 仍拒绝(L1 3.18/3.2)。前端不做"安全决定"。
- **敏感操作:** 平台侧 CA/KEK 等敏感操作的双人控制(L1 3.5/3.18)由后端流程强制,前端只呈现流程状态。
- **租户上下文:** 租户控制台的租户身份来自会话,前端不允许切换他租户;平台控制台访问租户数据需**理由记录**(L1 3.18),前端在入口收集理由并随请求上报。

依据:L1 3.18 "前端鉴权不可信、授权后端强制"——前端 RBAC 定位为体验收敛,不承担安全;这避免"前端漏判=越权"的错觉。
- 备选:前端做权限判定为主。落选:前端可被绕过,把安全寄托于前端是反模式(L1 3.18 已否)。

**风险** 前端权限元数据与后端不一致致 UI 显示了无权操作 → 操作仍被后端拒(安全不受损),元数据与后端角色定义同源(待确认接口);平台越权访问租户数据 → 后端理由记录 + 审计(L1 3.18),前端只采集理由。

**结论** 前端 RBAC 仅驱动 UI 可见/可用,授权后端强制、前端不可信;平台访租户数据前端采集理由、后端记审计。

### 3.5 数据层:server-state 为主

**背景** 控制台主要是 CRUD、列表、看板——以远端数据(server-state)为主,本地 UI 状态少。

**目标** 定数据获取/缓存/失效/刷新策略,避免重型全局状态。

**设计**
- **server-state(主):** 用数据获取/缓存库(候选 TanStack Query 类,3.8 待确认)管理远端数据的拉取、缓存、失效、重试;列表/详情/看板皆为 server-state。
- **client/UI state(最小):** 仅表单草稿、弹窗开合、筛选条件等本地状态;**不把远端数据塞进全局 store**。
- **刷新策略:** 用量看板/状态类(PoP 健康、PoP 推送)按需轮询或手动刷新;一般 CRUD 按操作后失效重取。**实时推送(WebSocket/SSE)起步不引入**(控制台非实时系统),待确认是否有强实时需求。
- **乐观更新:** 谨慎使用(写操作后以后端结果为准),避免与后端授权/校验结果不一致。

依据:控制台数据特征是 server-state 为主,用查询库 + 最小 client state 比重型全局 store(Redux 全家桶)更贴合、样板更少。
- 备选:全局 store(Redux 等)统管。落选:对 CRUD/列表为主的控制台过度设计,样板多、缓存/失效需自造(查询库已内置)。

**风险** 缓存失效不当致看到陈旧数据 → 写后失效相关查询 + 合理 staleTime;轮询过频增后端负载 → 看板按需/手动刷新 + 合理间隔。

**结论** server-state 用查询库(候选待确认)、client state 最小;写后失效重取;实时推送起步不引入;乐观更新谨慎。

### 3.6 关键交互设计点

**背景** 多数页面是常规 CRUD,但少数交互复杂、且直接承接控制面子 L2 的能力,需点名设计。

**目标** 列出关键交互及其设计要点与对应控制面子 L2。

**设计 —— 关键交互**

| 交互 | 要点 | 承接 |
|------|------|------|
| **策略编辑器** | 可视化**求值顺序**(默认拒绝 + 优先级首次匹配)、**遮蔽/冲突告警**(编译期检测结果回显)、effect(allow/deny/inspect)、subject 选择器(组/姿态) | 策略编译器子 L2(3.2 求值语义、RP2 可视化、LP6 冲突档、LP8 防抖)|
| **用量看板** | 席位(月活跃去重)、带宽用量与超量、趋势;租户侧只见本租户。**对客界面:自研轻量看板,数据走我方 billing/metrics API,不内嵌任何第三方**(可控+品牌一致+合规观感;区别于平台内部运维监控用自托管 Grafana,见 `sase-l2-frontend-platform-console.md` 3.2) | 计费计量子 L2(UsageRecord、C8/C9)|
| **审计日志查询** | 按 actor/action/target/时间过滤;只读;租户侧限本租户 | 控制面 `audit`(L1 3.3 AuditLog、3.14 留存)|
| **IdP 配置向导** | OIDC/SAML/SCIM + 企微/钉钉/飞书分流配置;secret 输入只写不回显(存 KMS 引用) | 身份与 IdP 子 L2(3.1 adapter、3.7 secret)|
| **站点/CPE 开通** | ZTP 引导、激活码、状态 | resource 模块 / SD-WAN CPE L2(**待产出**,前向衔接)|

- **策略编辑器**是最复杂交互:把编译器的"求值顺序/遮蔽/冲突"以可视化呈现,让租户管理员在编辑时即看到生效顺序与潜在遮蔽(承接编译器 RP2)。具体编辑器组件设计留实现期,本文定"需呈现求值顺序 + 遮蔽告警"的契约。
- **secret 输入**(IdP client_secret 等)**只写不回显**:前端提交后不读回明文(后端存 KMS 引用,L1 3.5),前端只显示"已配置"。

依据:这些交互直接关联安全语义(策略)、收入(用量)、合规(审计)、密钥(IdP secret),呈现错误会误导操作,故点名;常规 CRUD 不赘述。

**风险** 策略可视化与编译器实际语义不一致致误导 → 可视化以编译器输出(求值顺序/遮蔽结果)为准、不自行推算(承接编译器契约);secret 回显泄露 → 只写不回显(上)。

**结论** 点名五类关键交互;策略编辑器呈现求值顺序 + 遮蔽告警(承接编译器)、secret 只写不回显;常规 CRUD 不赘述。

### 3.7 横切:错误 / i18n / 可观测 / 传输安全

**背景** 跨所有页面的横切关注点需统一,避免散落。

**目标** 定错误处理、i18n、前端可观测、传输安全的取向。

**设计**
- **错误处理:** 统一拦截 API 错误(按后端错误契约 3.3),分类呈现(鉴权失效→重登、无权→提示、校验错→表单回显、服务错→重试/上报);全局错误边界兜底。
- **i18n:** **中文为主**(客群中国大陆),i18n 框架预留多语;文案集中管理。
- **前端可观测:** 前端错误/性能埋点(页面加载、API 失败率)上报;**不含租户敏感数据**;与后端遥测(L1 3.14)分离或对接,待确认。
- **传输安全:** 浏览器↔控制面走 HTTPS;**若强合规客户要求国密**,浏览器↔控制面(管理面/Admin API)的国密 TLS(TLCP)**由前置国密 TLS 终结/网关处理**,**前端应用本身算法无关**。该面已纳入国密选型 `sase-gm-crypto-selection.md` **3.8(v0.2,管理面前置国密终结)**;前置网关部署/高可用留实施期(LF7)。会话令牌存储与 CSRF/XSS 防护按后端约定(待确认)。

依据:横切统一避免各页面各写一套;传输国密是部署层(前置终结)而非前端应用层职责,避免把国密拖进前端。

**风险** 前端埋点含敏感数据 → 埋点白名单 + 脱敏;XSS/CSRF → 框架默认防护 + CSP + 令牌策略(待确认与后端协同);i18n 漏译 → 文案集中 + CI 检查缺失键。

**结论** 统一错误拦截 + 中文为主 i18n + 脱敏埋点 + HTTPS(国密由前置终结、前端算法无关);XSS/CSRF/令牌策略与后端协同(待确认)。

### 3.8 技术取向与待确认选型

**背景** 架构级取向可定;库级选型宜留待确认(并可对齐爱快既有前端栈)。

**目标** 定能定的取向,标清待确认的库级选型。

**设计 —— 已定取向**
- **SPA(非 SSR):** 控制台是登录后内部管理工具,无 SEO/首屏 SSR 需求 → SPA + 静态托管,部署简单。
  - 备选:SSR(Next 类)。落选:控制台无 SEO 需求,SSR 增服务端渲染复杂度与部署成本,收益不匹配。
- **React + TS**(L1 3.21 已定)。
- **OpenAPI 生成消费**(3.3)、**server-state 查询库**(3.5)、**双入口单库**(3.1)——架构取向已定,具体库待确认。

**待确认(库级选型,可对齐爱快既有前端栈):**
| 项 | 候选 | 说明 |
|----|------|------|
| 组件库/设计系统 | **Ant Design**(中文企业级生态成熟)候选;或对齐爱快既有 | 影响 UI 一致性与开发速度 |
| 数据获取/缓存 | TanStack Query 类 | server-state(3.5) |
| 路由 | React Router 类 | 双入口路由树 |
| 构建/打包 | Vite 类 | SPA 构建 |
| OpenAPI 代码生成 | openapi-typescript 类 | 生成类型/client(3.3) |
| 状态(client) | 轻量(Zustand 类)或内置 | UI state(3.5) |

依据:架构取向(SPA/React/生成消费/server-state/双入口)与项目特征明确可定;库级选型受**爱快既有前端栈**影响大,留待确认以便对齐(若有house栈则采用,无则按候选)。

**风险** 库选型与爱快既有栈不一致致团队学习成本 → 优先对齐既有栈;组件库锁定 → 经设计系统层封装,降低更换成本。

**结论** 取向已定(SPA + React/TS + OpenAPI 生成 + server-state + 双入口);组件库等库级选型待确认、优先对齐爱快既有前端栈。

## 四、风险

### RF1:把前端当安全边界
最易犯。缓解:前端 RBAC 仅体验层、授权后端强制(3.4,L1 3.18);设计与评审反复强调"前端不可信"。

### RF2:与 Admin API 契约漂移
缓解:OpenAPI 生成 + 编译期类型校验 + CI 校验(3.3);不手写请求类型。

### RF3:平台能力混入租户控制台(暴露面)
缓解:双入口物理隔离、平台能力不进租户包(3.1)。

### RF4:secret/敏感数据在前端泄露
缓解:secret 只写不回显(3.6)、埋点脱敏(3.7)、不缓存敏感明文。

### RF5:策略可视化误导(与编译器语义不符)
缓解:可视化以编译器输出为准、不自行推算(3.6,承接编译器契约)。

### RF6:库选型与爱快既有栈脱节
缓解:库级选型待确认、优先对齐既有栈(3.8);设计系统层封装降低锁定。

### RF7:国密客户访问控制台的传输合规
缓解:管理面国密 TLS 由前置终结(3.7),前端算法无关;**已补入国密选型 3.8(v0.2)**;前置网关部署/高可用留实施期(LF7)。

## 五、结论与衔接

**结论:** 前端控制台 = **单代码库 + 共享层 + `app-admin`/`app-tenant` 双入口**(平台能力不进租户包);模块/路由按业务域与控制面对齐;**以 OpenAPI 生成 TS 类型 + client 消费 Admin API**、CI 防漂移;**前端 RBAC 仅体验层、授权后端强制**(L1 3.18,前端不是安全边界);数据层 **server-state 为主**(查询库)+ 最小 client state;关键交互(策略编辑器求值可视化 + 遮蔽告警、用量看板、审计查询、IdP 配置 secret 只写不回显)承接对应控制面子 L2;横切统一错误/i18n(中文为主)/脱敏埋点/HTTPS(**国密由前置 TLS 终结、前端算法无关**);技术取向 **SPA + React/TS**,**库级选型待确认、优先对齐爱快既有前端栈**。

**衔接:**
- **控制面 `admin` 模块 / Admin API(OpenAPI):** 前端唯一数据来源,单一契约。
- **策略编译器子 L2:** 策略编辑器呈现求值顺序/遮蔽/冲突(RP2/LP6/LP8)。
- **计费计量子 L2:** 用量看板呈现 UsageRecord(C8/C9)。
- **身份与 IdP 子 L2:** IdP 配置向导(adapter 类型、secret 只写不回显)。
- **SD-WAN CPE L2(另文):** 站点/CPE 开通(ZTP)前端。
- **平台运维控制台功能深化(`sase-l2-frontend-platform-console.md`):** `app-admin` 入口的逐功能域设计 + 后端 API 契约(几乎全新建)——头号前置是跨租户数据路径(承接数据访问层 L2 `InPlatformTx`)+ OpenAPI 规格;真实第一刀在后端而非前端页面。
- **国密选型:** 管理面(浏览器↔控制面)国密 TLS 由前置终结(前端算法无关);**已补入国密选型 3.8(v0.2)**,前置网关部署留实施期(LF7/RF7)。
- 不依赖国密 PoC-G / PoC-1。

搭前端脚手架 / 写前端代码须**另行授权**(设计先行)。
