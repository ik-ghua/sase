## 一、背景

L1 3.2 把"跨租户泄漏 = 0"列为不可妥协约束,选定**共享 schema + Postgres RLS**,并明确两个泄漏点:① 新增表漏配 RLS;② 连接池跨租户串号。控制面 L2 总览把这些落到 `data` 横切层,定了规则 3(任何模块经 `data` 层、强制 RLS 上下文),但未给机制规格。

本文回答三个工程问题:**RLS 具体怎么配才不被绕过?租户上下文怎么注入才不串号?漏配怎么在 CI 拦死?** 这些是"0 泄漏"从设计承诺变成可验证实现的关键。

---

## 二、目标(可衡量)

1. **任一查询无法越权**:未带正确租户上下文的查询返回 0 行(fail-closed),不泄漏他租户数据。
2. **连接池无串号**:连接在租户间复用时,上一租户的上下文不残留到下一请求。
3. **漏配即失败**:任何含 `tenant_id` 的表若未启用并强制 RLS、或无策略,**CI 构建失败**(L1 3.2 风险①)。
4. **统一入口**:业务模块不得绕过 `data` 层直连 Postgres(总览规则 3),CI 强制。
5. **决定库选型**:DB 访问方式、迁移工具、依赖检查工具(解总览 LC2/LC3 中数据层相关项)。
6. **纵深防御**:应用层 bug 也不致越权——隔离由 DB 引擎兜底(L1 3.2)。

**非目标:** 字段级加密的密码学实现(只定边界,3.8);查询性能调优(索引等)留实现期;读写分离/分片(规模化议题)。

## 三、设计

### 3.1 隔离层次:DB 引擎 RLS + 应用层作用域(纵深)

**背景** L1 3.2 要求"应用 bug 也不越权"。单靠应用层 `WHERE tenant_id=?` 不够——漏一处即泄漏。

**目标** 明确两层隔离及各自职责,确立 DB 引擎为最终兜底。

**设计 —— 两层**
1. **DB 引擎层(最终兜底,强制):** Postgres RLS——即使应用层忘了带 `tenant_id` 条件,引擎按当前租户上下文过滤,越权查询返回 0 行。这是不可绕过的底线。
2. **应用层(纵深 + 可读性):** `data` 层在查询中仍显式带租户作用域,并在入口断言上下文已设置(fail-loud,尽早暴露 bug)。

依据:RLS 是兜底(漏写条件也不泄漏),应用层作用域是"尽早响亮失败"(catch bug)。两层职责不同:**RLS 保证不泄漏(fail-closed),应用层保证 bug 被发现(fail-loud)**。
- 备选:仅应用层作用域,不用 RLS。落选:漏一处即泄漏,违背 L1 3.2 "0 泄漏"硬约束。
- 备选:仅 RLS,应用层不带条件。落选:bug(如上下文未设)只表现为"查不到数据",难定位;且失去显式可审计性。

**风险** 两层都对则冗余条件 → 可接受(安全 > 微小冗余);RLS 与应用条件不一致致困惑 → 以 RLS 为准,应用条件须是 RLS 的子集或等价。

**结论** 双层隔离:RLS 引擎兜底(fail-closed,不泄漏)+ 应用层作用域(fail-loud,早暴露)。DB 引擎是最终防线。

### 3.2 RLS 策略定义

**背景** RLS 配置有几个易错点会使其形同虚设:表 owner 默认绕过 RLS、连接角色带 BYPASSRLS、未 FORCE。

**目标** 给出每张租户表的 RLS 配置模式,堵住"看似开了 RLS 实则被绕过"。

**设计 —— 每张含 `tenant_id` 的表统一模式(DDL 说明,非完整脚本)**

```sql
-- 1) 开启并【强制】RLS:FORCE 使表 owner 也受约束(否则 owner 绕过)
ALTER TABLE <t> ENABLE ROW LEVEL SECURITY;
ALTER TABLE <t> FORCE ROW LEVEL SECURITY;

-- 2) 策略:读写都按当前租户上下文过滤
--    missing_ok=true → 上下文未设时 current_setting 返回 NULL → tenant_id = NULL 为假 → 0 行(fail-closed)
CREATE POLICY tenant_isolation ON <t>
  USING      (tenant_id = current_setting('app.current_tenant', true)::uuid)
  WITH CHECK (tenant_id = current_setting('app.current_tenant', true)::uuid);
```

三个关键约束(缺一即被绕过):
1. **应用连接角色 ≠ 表 owner,且无 `BYPASSRLS` 属性、非 superuser。** owner/superuser/BYPASSRLS 角色绕过 RLS。落地:迁移用 owner/迁移角色;应用运行用受限角色——**读写用 `app_rw`,只读消费者(如 xDS server,见 xDS 子 L2)用 `app_ro`(SELECT-only)**;两者均非 owner、无 BYPASSRLS、非 superuser、均受 RLS 约束。
2. **`FORCE ROW LEVEL SECURITY`。** 即使将来误用 owner 连接,也受约束(纵深)。
3. **`WITH CHECK` 与 `USING` 同条件。** `USING` 管读/删/更新可见行,`WITH CHECK` 管插入/更新写入行——防止写入他租户 `tenant_id`。

依据:`missing_ok=true`(`current_setting` 第二参)使上下文缺失时返回 NULL 而非报错 → 比较为假 → **0 行,fail-closed,绝不泄漏**;响亮失败交给应用层(3.3 断言)。
- 备选:`current_setting('app.current_tenant')`(无 missing_ok)未设时**抛错**。取舍:抛错更响亮但使"上下文未设"成为运行时错误风暴;选 fail-closed(0 行)在 DB 层 + fail-loud(断言)在应用层,各取所长。

**风险** 新表漏掉此模式 → 3.7 CI 门禁拦截;策略写错(如漏 WITH CHECK)→ 隔离测试(3.9)覆盖写入越权。

**结论** 每张租户表:ENABLE+FORCE RLS + USING/WITH CHECK 同条件 + 受限应用角色;`missing_ok` 实现 DB 层 fail-closed。

### 3.3 租户上下文注入(事务级)

**背景** RLS 靠 `app.current_tenant` 运行期变量。怎么设、生命周期多长,直接决定会不会串号(L1 3.2 风险②)。

**目标** 选定上下文注入方式,使其**绑定事务、自动失效、不残留连接**。

**设计**
- **用 `set_config('app.current_tenant', <tenant_id>, true)`(第三参 `is_local=true`)= 事务级(等价 `SET LOCAL`)。** 事务结束(COMMIT/ROLLBACK)自动失效,**不随连接归还池而残留**。
- **强制事务包裹:** 所有租户作用域的访问(含只读)都在事务内执行——`data` 层 acquire 连接 → `BEGIN` → `set_config(...,true)` 为首条语句 → 业务查询 → `COMMIT`/`ROLLBACK`。
- **链路:** HTTP/gRPC 中间件从认证主体解析 `tenant_id` → 放入 `context.Context` → `data` 层开事务时读 ctx 注入;**ctx 无 tenant_id 时 `data` 层直接报错(fail-loud,3.1 应用层职责)**,不发查询。
- **平台级访问**(无租户)走单独路径(3.6),不注入该变量。

依据:事务级 `SET LOCAL` 是消除连接池串号的**结构性**办法——不依赖"记得在归还时重置",事务边界即上下文边界。
- 备选:会话级 `SET app.current_tenant`(非 LOCAL)+ 连接归还时 `RESET`/`DISCARD`。落选:依赖每次归还显式重置,**漏一次 = 串号**(正是 L1 3.2 风险②);且连接池复用时存在窗口。事务级从机制上免除此风险。
- 备选:每查询参数显式传 `tenant_id`、不用运行期变量。落选:那样 RLS 无法工作(RLS 策略需运行期变量),退化为纯应用层(3.1 已论证不足)。

**风险** "强制事务包裹只读"带来轻微开销 → 可接受(安全优先);忘记包裹事务 → `data` 层 API 设计上**只暴露事务内执行入口**(3.5),从 API 形态上杜绝裸查询。

**结论** 事务级 `set_config(...,true)` 注入,绑事务自动失效;中间件→ctx→事务链路;ctx 缺租户即 fail-loud;从 API 形态强制事务包裹。

### 3.4 连接池语义与重置

**背景** 控制面用连接池(pgxpool 候选)。池化连接跨请求复用,是串号的物理温床。

**目标** 明确池配置与重置语义,与 3.3 配合保证零残留。

**设计**
> 措辞细化:总览 3.4 写"连接池归还**必须重置**";本文据事务级机制把"重置"由**必须**降为**纵深兜底**(主保证改为事务级自动失效)——是机制深化,非与总览冲突。

- **主机制:3.3 的事务级上下文** → 连接归还池时,事务已结束,`app.current_tenant` 已自动失效,**无残留**。这是零串号的主保证。
- **兜底:连接归还/获取时重置。** 池的 `AfterRelease`/`BeforeAcquire` 钩子(pgxpool 支持)执行 `RESET ALL` 或 `DISCARD ALL` 作为第二道保险——即便将来误用会话级设置也清掉。
- **禁会话级设置:** 代码层禁止 `SET`(非 LOCAL)`app.current_tenant`;依赖检查(3.7)可加规则拦截。
- **连接角色:** 池统一用受限 `app_rw` 角色(3.2 约束 1),不用 owner/superuser 连接。

依据:主机制(事务级)已结构性消除残留,兜底(钩子重置)是纵深;两者叠加使"串号"需同时突破两层才可能。

**风险** 连接级 `prepared statement`/临时表残留 → `DISCARD ALL` 覆盖;长事务占用连接 → 事务尽量短(只读也短包裹),监控长事务(对接 L1 3.14)。

**结论** 事务级上下文(主)+ 归还重置钩子(兜底)+ 禁会话级设置 + 受限角色;连接池零残留靠机制而非纪律。

### 3.5 数据访问 API 与库选型(解 LC2)

**背景** 总览 LC2 留了 DB 访问方式(sqlc / ORM)与迁移工具未定。数据层 API 形态还要承载"强制事务包裹"(3.3)与"不可绕过"(规则 3)。

**目标** 选定 DB 访问库、迁移工具,定 `data` 层对业务模块的 API 形态。

**设计 —— 选型决策**

| 项 | 决策 | 依据 | 备选及落选 |
|----|------|------|-----------|
| DB 访问 | **sqlc + pgx** | 编译期校验 SQL、生成类型安全 Go;**SQL 显式可审计**(RLS 场景关键,能逐条看清查询);对连接/事务**显式掌控**(3.3 强制事务必需) | **ORM(GORM/ent)**:隐藏查询、易绕过 `data` 层、对事务/连接掌控弱、迁移与 RLS 控制隐晦 → 落选;**raw database/sql**:缺类型安全、易写错 → sqlc 已覆盖其优点 |
| 迁移工具 | **golang-migrate**(SQL up/down) | 纯 SQL、简单、CI 友好;RLS DDL(3.2)直接写迁移 | **goose** 等价(可选);**atlas**(schema-as-code、可 lint)更强但引入依赖/学习成本 → 待评估(附录) |
| 依赖检查 | **depguard**(CI) | 规则 1:禁 `data` 包外 import `pgx`/`database/sql`(强制规则 3 不绕过);规则 2:禁会话级 SET(配合 3.4) | go-arch-lint 偏架构分层校验,与 depguard 互补,**待与总览 3.3 规则 2 一并定** |

**`data` 层 API 形态(强制事务、不可绕过):**
- 对业务模块只暴露**事务内执行入口**,形如 `data.InTx(ctx, func(q *Queries) error {...})`:`InTx` 内部 acquire 连接 → BEGIN → 注入租户上下文(3.3)→ 执行回调 → COMMIT/ROLLBACK。业务模块拿到的 `*Queries`(sqlc 生成)只能在事务内用。
- **只读入口 `data.InTxRO(ctx, fn)`:** 与 `InTx` 同构,但用 `app_ro` 角色(SELECT-only)、单租户上下文、只读事务。供**只读消费者**(如 xDS server 按租户读资源,见 xDS 子 L2 3.3)使用——仍是**单租户/事务级**,多租户读=按租户多次 `InTxRO` + 调用方内存聚合,不破坏单租户 RLS 语义。
- **不暴露**裸连接/裸池给业务模块 → 从 API 形态上杜绝"忘了包事务"或"忘了设上下文"。
- 平台级访问用单独入口(3.6),显式不注入租户上下文。

依据:把"强制事务 + 强制上下文"做进**唯一入口**,使正确用法是唯一可用法(misuse-resistant),而非靠文档约定。

**风险** sqlc 对极动态查询(可变过滤)支持有限 → 少量动态查询用 pgx 手写但仍经 `data` 层、仍在 `InTx` 内;生成代码与 SQL 漂移 → CI 校验生成物(对接总览 3.6)。

**结论** sqlc + pgx + golang-migrate + depguard;`data` 层只暴露事务内入口 `InTx`,misuse-resistant;动态查询经 `data` 层手写兜底。

### 3.6 平台级表与豁免

**背景** L1 3.3 有非租户作用域的平台级表(如 `PoP`)。它们不该带租户 RLS,但又不能成为"漏配 RLS"的误报或绕过缺口。

**目标** 明确平台级表的访问路径与豁免标注,使门禁(3.7)能区分"故意豁免"与"漏配"。

**设计**
- **判定:** 表**不含 `tenant_id` 列**即平台级(如 `PoP`)。门禁(3.7)只对**含 `tenant_id`** 的表要求 RLS;不含的天然在范围外。
- **显式豁免清单:** 维护一份平台级表白名单(如 `migrations` 元数据或独立清单),门禁交叉核对:含 `tenant_id` 却想豁免的表**必须在白名单且写明理由**,否则失败——防止"加了 tenant_id 又故意不配 RLS"的灰色绕过。
- **访问路径分离:** 平台级访问走 `data` 层单独入口(`InPlatformTx`),用平台角色,**不注入 `app.current_tenant`**;平台级操作权限由 `authz`(平台 RBAC,L1 3.18)控制,而非 RLS。
- **跨域表(少数):** 若某表既有平台视角又有租户视角(暂未识别),列入待确认(附录),不在本版臆造规则。

依据:用"是否含 `tenant_id`"作机械判定,辅以白名单堵灰色地带,使门禁无误报且无绕过。
- 备选:**所有表一律配 RLS,平台表用恒真空策略**。落选:平台表查询需额外恒真条件、语义晦涩,且"恒真策略"易与"漏配后误开"混淆,反增灰色地带 → 不如显式判定 + 白名单清晰。
- 备选:**用专门标注列(如 `is_platform`)代替"`tenant_id` 缺失"判定**。落选:多一列冗余且可漏标;"是否含 `tenant_id`"本就是租户作用域的充要标志,无需额外列。

**风险** 平台级表误带 `tenant_id` → 门禁要求其配 RLS 或进白名单说明,暴露设计问题(好事);白名单滥用 → 白名单变更需 review + 理由(对接审计)。

**结论** 含 `tenant_id` 即受 RLS 门禁;平台级表走独立入口 + 平台 RBAC;白名单(**条目须含豁免理由**)堵"加 tenant_id 又不配 RLS"的灰色绕过。

### 3.7 漏配 RLS 的 CI 门禁

**背景** L1 3.2 风险① 要求"新增表漏配 RLS → 构建失败"。这是"0 泄漏"最容易回归的点。

**目标** 给出可在 CI 跑的机械检查,使漏配/错配无法合入。

**设计 —— 针对 Postgres catalog 的断言测试(在迁移后的测试库上跑)**

对每张表(查 `pg_class` / `pg_policies` / `information_schema.columns`)断言:
1. **含 `tenant_id` 列的表** 必须:`relrowsecurity = true`(ENABLE)**且** `relforcerowsecurity = true`(FORCE)**且** 存在按 `app.current_tenant` 过滤的策略(USING 与 WITH CHECK 均有)。
2. **不含 `tenant_id` 的表** 必须在平台级白名单(3.6),否则报"疑似漏标租户列或漏豁免"。
3. **应用角色 `app_rw` 与只读角色 `app_ro`** 均无 `BYPASSRLS`、非 superuser、非各表 owner(查 `pg_roles` / 表 owner);`app_ro` 仅授 `SELECT`(无写权限)。

落地:作为 CI 必过测试(`make test` 的隔离子集),失败即构建红。
- 备选:用 atlas/sqlc 的 schema lint 静态检查。**暂不选**,理由:catalog 断言**在真实迁移后的库上验证**最贴近运行态(连 FORCE、角色属性都能查),起步用它;静态 lint 待评估(附录 LD1)。

**风险** 测试库与生产 schema 漂移 → 测试库由同一套 `migrations/` 构建(总览 3.5),保证一致;新增"含 tenant_id 但合理无需 RLS"的特例 → 走 3.6 白名单 + 理由。

**结论** CI 在迁移后测试库上查 catalog,断言含 `tenant_id` 表的 ENABLE+FORCE+策略、角色无 BYPASSRLS、平台表在白名单;失败即红。这是"0 泄漏"的回归防线。

> **✅ as-built(Slice32):** catalog 门禁 G1/G2/G3 已编码 `internal/data/rls_catalog_gate_test.go`(数据驱动查 pg_class/pg_roles/pg_policy:G1 含 tenant_id 基表必 ENABLE+FORCE+策略 + 非空性断言防空过 + tenants[id 键]单独断言;G2 app_rw/app_ro/app_platform_ro 均无 BYPASSRLS/superuser;G3 平台视图 `tenant_summary` 存在+是视图+owner 绕 RLS,与 migration 0013 owner 自检呼应)。**G4(隔离行为)** 由 `rls_integration_test.go`(越权读/写/上下文缺失/连接复用)+ `platform_integration_test.go`(平台路径)覆盖。go-code-reviewer 审过(catalog SQL 正确、防空过到位)。

### 3.8 字段级加密集成边界

**背景** L1 3.2/3.5 要求每租户 DEK 信封加密敏感数据(如 `IdPConfig.secret_ref` 指向的秘密、可能的 PII 列),支持密钥销毁式删除(3.16)。加密在 `secret` 模块,但落库经 `data` 层。

**目标** 只定 `data` 层与 `secret` 模块的边界,不深挖密码学(另子 L2)。

**设计**
- **边界:** `data` 层负责"密文列的读写";`secret` 模块负责"用租户 DEK 加解密"(DEK 由 KEK 包裹,KEK 在 HSM/KMS,L1 3.5)。
- **形态:** 敏感列存密文 + 关联的密钥版本/DEK 引用;写时 `data` 层调 `secret.Encrypt(tenant_id, plaintext)`,读时 `secret.Decrypt(...)`。哪些列加密由模型标注(待 `secret` 子 L2 定清单)。
- **与 RLS 的关系:** 加密是"即便越权拿到行也读不到明文"的**额外**纵深;RLS 仍是不泄漏行的第一层。两者正交叠加。
- **密钥销毁式删除:** 注销租户删 DEK → 该租户密文不可恢复(L1 3.16);`data` 层不持明文、不缓存 DEK。

依据:职责分离——`data` 管行级隔离与密文存取,`secret` 管密钥与算法;避免把密码学散进数据层。

**风险** 加密列无法用 SQL 过滤/索引 → 仅对确需保密的列加密(清单待定),不滥用;DEK 调用增延迟 → 缓存解包后的 DEK(短时、内存、随租户上下文),**缓存策略待 `secret` 子 L2 定**。

**结论** `data` 管密文列读写、`secret` 管 DEK 加解密,正交于 RLS;加密列清单与 DEK 缓存留 `secret` 子 L2;`data` 层不持明文/DEK。

### 3.9 隔离测试

**背景** L1 3.20 把"跨租户 0 泄漏"列为 CI 验收。3.7 验证 RLS **配置**存在,本节验证 RLS **行为**正确。

**目标** 定自动化跨租户泄漏测试用例集,纳入 CI 门禁。

**设计 —— 测试用例类**
1. **越权读:** 设租户 A 上下文,查应只见 A 的数据,断言查不到 B 的行(即便用 B 的 id 显式查)。
2. **越权写:** 设 A 上下文,尝试插入/更新 `tenant_id=B` 的行 → 被 `WITH CHECK` 拒绝。
3. **上下文缺失:** 不设上下文执行查询 → 返回 0 行(fail-closed,3.2),且 `data` 层入口断言报错(fail-loud,3.3)。
4. **连接复用:** 同一池连接先服务 A 再服务 B,断言 B 的查询绝不返回 A 残留(验证 3.3/3.4)。
5. **平台级:** 平台入口可跨租户读 `PoP` 等;但平台入口不能读租户表(除非显式),验证路径分离(3.6)。

落地:作为 L1 3.20 "隔离测试"子集,**覆盖率纳入门禁**(漏测=假安全)。
- 依据:配置正确(3.7)≠ 行为正确(策略可能写错条件);行为测试补上这层。

**风险** 用例覆盖不全 = 假安全 → 每新增租户表,模板化生成其越权读写用例(随表清单驱动);测试数据构造繁琐 → 测试夹具按租户造数。

**结论** 五类越权/复用/缺失用例入 CI,覆盖率门禁;配置门禁(3.7)+ 行为测试(3.9)共同守"0 泄漏"。

### 3.10 超大租户迁库逃生

**背景** L1 3.2 给共享 schema 留了逃生通道:超大租户后续迁独立库,`tenant_id` 模型使其为"数据迁移"而非"schema 改造"。本节落实数据层如何支持。

**目标** 明确逃生路径,使迁库对业务模块尽量透明。

**设计**
- **路由层:** `data` 层按 `tenant_id` 决定连接目标(默认共享库;白名单租户指向其独立库)。路由表小、变更低频。
- **迁移:** 独立库 schema 与共享库一致(同 `migrations/`),迁数据按 `tenant_id` 导出导入;迁移期双写/校验窗口(详细流程**待运维子 L2 / 实施期**)。
- **RLS 仍在:** 独立库内同租户(单租户)RLS 仍开(纵深不变),`app.current_tenant` 注入逻辑不变。
- **业务透明:** 因业务模块只经 `data` 层 `InTx`(3.5),路由对其透明,无需改业务代码。

依据:`tenant_id` 一致 + 统一 `data` 入口,使迁库收敛在路由层;这是 L1 "为数据迁移而非 schema 改造"承诺的落点。

**风险** 跨库事务/外键不可用 → 单租户数据自洽(都属同租户),无跨库事务需求;路由错误指向 → 路由变更走审计 + 迁移校验。

**结论** 迁库 = `data` 层按 `tenant_id` 路由 + 同 schema 数据迁移,对业务透明,RLS 不变;详细迁移流程留实施期。

## 四、风险

### RD1:RLS 被角色配置绕过(owner/BYPASSRLS/superuser)
最隐蔽的失效。缓解:FORCE RLS(3.2)+ 应用用受限 `app_rw` 角色 + CI 断言角色无 BYPASSRLS/非 owner(3.7)。

### RD2:连接池串号(L1 3.2 风险②)
缓解:事务级上下文自动失效(3.3,主)+ 归还重置钩子(3.4,兜底)+ 禁会话级 SET + 连接复用测试(3.9 用例 4)。

### RD3:新表漏配 RLS(L1 3.2 风险①)
缓解:CI catalog 断言(3.7),含 `tenant_id` 表漏配即构建红;平台表白名单(3.6)防灰色绕过。

### RD4:策略写对了配置但写错了条件
配置存在 ≠ 行为正确。缓解:隔离行为测试(3.9)覆盖越权读/写/缺失。

### RD5:绕过 `data` 层直连 Postgres
缓解:depguard 禁 `data` 包外 import pgx/database/sql(3.5);`data` 层不暴露裸连接。

### RD6:fail-closed 掩盖 bug(上下文未设默默返回 0 行)
缓解:应用层 `data` 入口断言上下文存在 → fail-loud(3.1/3.3),不让"查不到"静默。

## 五、结论与门禁清单

**结论:** 数据访问层以 **DB 引擎 RLS(fail-closed 兜底)+ 应用层作用域(fail-loud 早暴露)** 双层隔离;租户上下文用**事务级 `set_config(...,true)`** 注入、绑事务自动失效,从机制上消除连接池串号;`data` 层用 **sqlc+pgx**、只暴露事务内入口 `InTx`(misuse-resistant)、业务模块不可绕过(depguard);平台级表按"无 `tenant_id`"判定 + 白名单;**漏配/错配靠 CI catalog 断言(配置)+ 隔离行为测试(行为)双门禁拦死**;超大租户迁库收敛在 `data` 层路由。

**CI 门禁清单(守 L1 "0 泄漏"):**
| 门禁 | 检查 | 来源 |
|------|------|------|
| G1 RLS 配置 | 含 `tenant_id` 表 ENABLE+FORCE+策略(USING/WITH CHECK) | 3.2/3.7 |
| G2 角色 | `app_rw` 与 `app_ro` 均无 BYPASSRLS、非 superuser、非 owner;`app_ro` 仅 SELECT | 3.2/3.7 |
| G3 平台表 | 不含 `tenant_id` 或在白名单(含理由) | 3.6/3.7 |
| G4 隔离行为 | 越权读/写/上下文缺失/连接复用/平台路径 五类用例 | 3.9 |
| G5 不绕过 | depguard 禁 `data` 包外直连 DB、禁会话级 SET | 3.4/3.5 |

**下一步:** 本子 L2 可与 `secret` 子 L2(加密列清单 + DEK 缓存,承接 3.8)、`policy` 编译器子 L2(其 PolicyBundle 落库经本层)衔接。搭 monorepo / 写代码须另行授权(设计先行)。
