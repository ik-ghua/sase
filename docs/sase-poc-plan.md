## 一、背景与总目标

设计已到"PoC 墙":再往上叠 L2(PoP 单机编排、客户端/CPE 加密栈)与回写 L1 v0.5,都依赖两个尚未实测/验证的假设。本计划把它们设计成**可执行、有明确判据**的实验。

**总目标:** 对两个假设给出证据驱动的结论;PoC 通过→固化进选型/L1(v0.5)并解锁对应 L2;不通过→按预案调整方案(各 PoC 内列)。

**非目标:** 写产品代码/搭 monorepo(PoC 代码一次性、隔离在 `poc/` 目录,不进产品);压测到生产规模(PoC 验"可行性/量级",规模化另测)。

## 二、PoC-G:国密数据面 / TLS 性能验证

**背景** 国密选型(v0.2)结论"待 PoC 复核":SM4 无 ChaCha20 那种"免硬件加速即快"的特性,TLCP 比标准 TLS 重——性能是最大未知,直接决定 10Gbps/PoP(L1 2.2)与 P95<30ms(L1 2.2/3.19)在国密下是否成立。

**目标** 实测 SM4/TLCP 性能,对"国密下数据面/TLS 性能可接受"给出结论。

**假设(待验)** 在带 SM4 硬件加速的国密 CPU 上,SM4-GCM 吞吐与 TLCP 延迟使 10Gbps/PoP 与覆盖区 P95<30ms **仍成立**(或给出差距量级)。

**指标与判据**

| 编号 | 测什么 | 判据(pass) |
|------|--------|------------|
| M-G1 | SM4-GCM 单核吞吐(含**带/不带 SM4 硬件加速**对比 ChaCha20 基线) | 带加速时单核吞吐达到"使单 PoP ≥10Gbps 所需核数 ≤ 同档 ChaCha20 的合理倍数"(倍数阈值评审定);给出确切倍数 |
| M-G2 | TLCP 握手延迟(SM2 签/验/KX)+ 记录层延迟增量 | 加密各段延迟之和 ≪ 网络 RTT(<20ms 主导,L1 3.19);不破坏覆盖区 P95<30ms 预算 |
| M-G3 | 铜锁-Envoy 国密路径吞吐与稳定性(ext_authz + ext_proc 同启) | 功能正确 + 吞吐/稳定性达可用;否则回退前置终结代理(选型 3.4b) |
| M-G4 | TLCP-over-QUIC 可行性 | 能跑通则用;不成熟则 connector 通道退 TLCP-over-TCP(选型 3.2) |
| M-G5 | 控制信令面国密 mTLS 开销(xDS/控制通道,非热路径) | 开销可接受(非每包路径,阈值宽) |

**方法**
- M-G1:Go microbench,SM4-GCM(经铜锁 cgo / 国密库)vs ChaCha20-Poly1305;同机分别在"启用/禁用 SM4 指令"下跑,出每核 Gbps。
- M-G2:TLCP 握手 microbench(SM2 操作计时)+ 单连接 RTT 测量。
- M-G3:铜锁-Envoy 构建(或现成镜像)+ 简单上游,加载 ext_authz/ext_proc,压吞吐/观稳定性。
- M-G4:用支持国密的 QUIC 库尝试 TLCP-over-QUIC,记录可行性与坑。
- M-G5:国密 mTLS 连接建立 + 流式开销 microbench。

**环境(见四)** 代表性国密 CPU(带/不带 SM4 指令各一)、Linux、铜锁/GmSSL。

**不通过预案** SM4 吞吐显著不足→评估国密硬件加速卡/加密机卸载、或调整单 PoP 容量假设(回写 5.1);铜锁-Envoy 不稳→前置终结代理;QUIC 国密不行→退 TCP。

**产出** M-G1~M-G5 实测数 → 固化/调整选型结论 → 回写 L1 v0.5(隧道/PKI/TLS)。

## 三、PoC-1:共享 Envoy ↔ 租户 namespace 隔离验证

**背景** L1 3.7 方案 B(共享 Envoy 承载多租户)的隔离,文档一直标"以 PoC 验证为准、兜底为待证明的设计目标"。这是"跨租户 0 泄漏"(L1 2.2 硬约束)在数据面 L7 侧的命门。

**目标** 验证"共享 Envoy + per-tenant 内核机制"能否强制租户隔离,且**即便 L7 路由配错,内核兜底仍不越租户**。

**假设(待验)** per-tenant network namespace + VRF + veth + per-tenant routing table + TPROXY + SO_MARK/ip rule 的协同接线,能使共享 Envoy 的每条上游连接落在正确租户路由域;**L7 路由出错时,内核 namespace/路由域兜底,流量不突破到他租户**。

**指标与判据**

| 编号 | 测什么 | 判据(pass) |
|------|--------|------------|
| I-1 | 正常路径隔离正确性 | 租户 A 的流量只到 A 的域/上游,**跨租户可达性 = 0**(自动化跨租户探测全失败) |
| I-2 | 出口绑定 | Envoy 上游连接经 SO_MARK + ip rule **确实选中该租户路由表**;namespace 间无路由可达 |
| I-3 | **内核兜底(关键)** | **故障注入**:故意把 Envoy 的 L7 路由配成指向他租户上游 → 流量**仍被内核 namespace/路由域拦住、不越租户**(证明"兜底"假设,而非当前的"待证明") |
| I-4 | namespace 规模与开销 | 单机数千租户 namespace 的创建/内存/转发开销量级(关联 L1 M3) |
| I-5 | 转发性能开销 | TPROXY + per-tenant routing 相对直连的转发开销量级 |

**方法**
- 用 Linux netns + veth + VRF + per-tenant routing table + `ip rule` + TPROXY + `SO_MARK` 搭**最小多租户拓扑**(2–3 租户起,I-4 再扩量)。
- 起共享 Envoy(多租户 per-tenant listener/route/cluster 配置),上游用打标的回显服务。
- I-1/I-2:自动化跨租户可达性 + 路由表命中验证。
- **I-3:故障注入**——改 Envoy 路由指向他租户 cluster,验证内核是否兜底(这一步决定"方案 B 兜底"是真结论还是要降级)。
- I-4:脚本批量建 namespace 测上限/开销;I-5:转发吞吐/延迟对比。

**环境(见四)** Linux + root(netns/VRF/TPROXY/iptables、可能 eBPF)、Envoy。**macOS 跑不了。**

**不通过预案** 若内核兜底不成立(I-3 失败)→ 隔离不能只靠 L7 配置正确性 → 升级为"高隔离租户用方案 A 每租户独立 Envoy"(选型逃生通道,L1 3.7),或重新设计接线;结论回写 L1 3.7/3.2 与 PoP 单机编排 L2 前置。

**产出** I-1~I-5 结论 → 固化/降级 L1 3.7 方案 B + 3.2 隔离语义 → 解锁 PoP 单机编排 L2。

## 四、执行环境与前置(关键阻碍)

**背景** PoC 的价值是**真实环境实测数**;当前开发机为 **macOS**,无法直接执行。这是必须先解决的前置。

**阻碍与需求**

| PoC | 需要的环境 | 当前 macOS 能否 | 谁来提供 |
|-----|-----------|----------------|---------|
| PoC-G | 代表性**国密 CPU**(海光/兆芯/ARMv8.4 等,**带 SM4 指令一台 + 不带一台对比**)+ Linux + 铜锁/GmSSL | 否(SM4 数在非目标 CPU 无意义) | **待业务方提供测试机/云主机**(C-G1 联动) |
| PoC-1 | **Linux + root**(netns/VRF/TPROXY/iptables、可能 eBPF)+ Envoy | 否(macOS 无 Linux 网络栈) | **待提供 Linux 主机/特权容器** |

**建议(请决策)**
1. **harness 代码我现在可写**(PoC-G 的 Go benchmark、PoC-1 的 netns/VRF/TPROXY 脚本 + Envoy 配置、自动化测试),隔离在 `poc/` 目录;**执行在目标环境上真跑、取数**。
2. **环境获取**:① PoC-1 — 一台 Linux 云主机/裸机(root)即可起步;② PoC-G — 需排到代表性国密 CPU(与 M1 硬件选型、C-G1 合并);带/不带 SM4 加速各一以出对比。
3. **数据纪律**:harness 输出原始数 + 环境指纹(CPU 型号/是否 SM4 指令/内核版本),**我据真实输出写结论,不代填、不外推编造**。

**绝不做:** 在 macOS 上跑出数字冒充目标硬件结论;无环境时编造 PoC 结果。

## 六、风险

### RP-1:无目标环境致 PoC 停在"计划"
缓解:本文先就绪 + harness 可先写;环境获取列为前置决策(四);PoC-1 环境门槛低(一台 Linux root 即可),可先行。

### RP-2:在非目标硬件出数误导
缓解:数据带环境指纹;非目标硬件的数仅作"趋势参考"明确标注,不作判据结论(尤其 SM4 加速强相关)。

### RP-3:PoC 代码被误当产品骨架
缓解:隔离 `poc/`、一次性、不进产品 monorepo;结论提取后可弃。

### RP-4:PoC-1 故障注入(I-3)不通过
缓解:这正是要验的——不通过则按预案降级方案 A(L1 3.7 逃生通道),是有价值的负结论,非失败。
