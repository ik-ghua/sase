## 一、背景

L1 已在系统层定了部署(3.13)、可观测(3.14)、容灾(3.15)的方向,各组件 L2 也散落引用"IaC/发布/drain/DR/门禁/实施期"。但**这些没被收口成一份"可运营"的实施层设计**——谁怎么部署、怎么灰度发布而不断流、怎么监控/告警/守 SLO、怎么容灾、CI 卡哪些门禁、怎么合规运营。本文把它们收口为运维/部署 L2 总览。

**这是"能跑且可运营"的前提**:再好的架构,没有可重复部署、安全发布、可观测、可容灾、CI 门禁,就无法对外提供 SaaS。

---

## 二、目标(可衡量)

1. **可重复部署**:控制面 + PoP 经 IaC 一键、小时级增删(L1 3.13);配置/IaC 入 Git。
2. **安全发布**:控制面滚动、PoP 配置 xDS 灰度(分批+ack+回滚)、PoP 二进制分批+drain、客户端灰度——**发布不断流**。
3. **可观测 + SLO**:Metrics/Logs/Tracing 全租户打标;守 SLO(控制面 99.9% / 数据面 99.99% / 覆盖区 P95<30ms,L1 2.2);错误预算 + 告警;租户只见本租户。
4. **可容灾**:控制面 PITR + 跨区副本(RPO<5min/RTO<1h,C7);数据面多 PoP 切换;KEK/CA HSM 备份 + 双人控制;DR 演练。
5. **CI 门禁汇总**:把各 L2 的门禁(0 泄漏隔离/RLS/策略/依赖/契约/性能/国密)汇成一张发布门禁表。
6. **合规运营**:审计 append-only/留存(C13)、牌照+等保并行(R1)、配额/计量运营、租户生命周期(3.16)。

**非目标:** 云厂商/region 选定、硬件型号、IaC/代码本体、SecOps 引擎、字节级指标清单。

## 三、设计

### 3.1 部署形态与拓扑

**背景** 控制面与 PoP 负载特征不同(控制面非热路径、PoP 数据面贴内核),部署形态应分别定。

**目标** 定控制面/PoP/客户端的部署形态与拓扑。

**设计**
- **控制面**:中心 **K8s** 单区域 + 灾备(L1 3.13);Postgres 主从、Redis、ClickHouse 等;非热路径、单区域起步可接受,DR 见 3.5。
- **PoP**:起步 4 个(沪/京/深/成都,L1 3.13)、云区域 + BGP 多线 IP;**IaC 一键单元**、新增小时级;每 PoP 多节点 **N+1** + 健康检查。
- **PoP 部署形态(倾向 systemd/裸机,待实测)**:数据面贴内核(XDP/eBPF)、需 CAP_NET_ADMIN/BPF、管理 per-tenant netns——**倾向原生进程(systemd)或特权容器**;**PoC-1 已证特权容器内可建 netns + 跑共享 Envoy + SO_MARK**,但 **XDP/eBPF map 在容器内完整性 + 裸机 perf 权衡待实测**(PoP L2 LO1)。控制面 K8s、PoP 数据面可不强求容器化。
- **客户端/CPE 分发**:Agent 多平台分发(控制台下发/MDM)、CPE 软件 ZTP(客户端 L2/CPE L2)。
- **不硬编码 IP**(L1 3.13):客户端经域名/控制面取节点,云 BGP IP 迁自建时无需改端。

依据:控制面/PoP 负载特征不同 → 形态分治(K8s vs systemd/裸机);PoP 形态有 PoC-1 证特权容器可行、但 perf/eBPF 完整性待实测,故"倾向 + 待实测"。
- 备选:PoP 也上 K8s。落选(起步):数据面贴内核 + XDP/eBPF + netns 管理在 K8s 内复杂、perf 不确定;systemd/裸机更直接(待实测复核)。

**风险** PoP 形态未定影响自动化 → LO1 待实测拍板;控制面单区域 → DR(3.5);云 BGP IP 迁自建 IP 变更 → 不硬编码 IP。

**结论** 控制面 K8s+DR;PoP IaC 一键 + N+1 + BGP 多线,**形态倾向 systemd/裸机(待实测)**;客户端/CPE 分发 + ZTP;不硬编码 IP。

### 3.2 IaC 与环境

**背景** L1 3.13:PoP IaC(Terraform/Ansible)一键。需定 IaC 与环境管理。

**目标** 定 IaC 工具与环境/配置管理。

**设计**
- **IaC**:**Terraform**(云资源:VM/网络/BGP IP)+ **Ansible**(主机配置/部署 PoP 组件)(L1 3.13);**配置/IaC 入 Git**(版本化、审计、回滚)。
- **多环境**:dev/staging/prod 分离;prod 变更经评审 + 灰度。
- **配置分层**:平台配置(IaC/Git)与**租户配置(经控制面 + xDS 下发**,非 IaC)分清——租户配置走产品控制面(数据访问层/xDS),不走 IaC。
- **机密**:IaC 不存明文机密;机密入 KMS/HSM(L1 3.5)、Vault 等(待确认)。

依据:Terraform+Ansible 是 L1 已定方向;IaC 入 Git 给版本化/审计/回滚;平台配置 vs 租户配置分层(租户配置是产品功能、不是基础设施)。
- 备选:手工部署。落选:不可重复、不可审计、小时级增删做不到(L1 3.13)。

**风险** IaC 漂移(手改导致与 Git 不符)→ 禁手改 + 周期对账;机密管理 → KMS/HSM + 不入 Git。

**结论** Terraform(云资源)+ Ansible(配置)、入 Git 版本化;dev/staging/prod 分离;平台配置走 IaC、租户配置走控制面;机密入 KMS/HSM。

### 3.3 发布与灰度

**背景** L1 3.13:控制面滚动、PoP 配置 xDS 灰度、数据面分批+drain;发布不能断流。

**目标** 定各层发布方式,保证灰度 + 回滚 + 不断流。

**设计 —— 分层发布**
- **控制面(K8s)**:滚动升级 + 健康检查;DB schema 迁移经 `migrations/`(数据访问层 L2)+ 向后兼容窗口。
- **PoP 配置(经 xDS)**:per-PoP **灰度**(分批/per-PoP 作用域 xDS L2 3.4 + 版本/ack/NACK 回退 3.8);坏配 PoP 保留旧版不停摆(xDS NACK)。
- **PoP 二进制**:分批升级 + **优雅 drain**(在途连接靠客户端重连,L1 3.13;PoP L2 3.9);健康检查 + 自动回滚。
- **客户端 Agent/CPE**:**灰度升级 + 回滚**(分批放量、监控、可回退,客户端 L2 LA8/CPE L2 LS8);版本兼容(协议版本字段,L1 3.11)。
- **回滚**:配置回滚=推上一版(xDS);二进制回滚=分批回退;策略回滚=激活上一 PolicyBundle(编译器 L2)。

依据:分层发布(控制面/配置/二进制/客户端)各用合适机制 + 灰度 + ack/健康检查 + 回滚,兑现 L1 3.13"发布不断流";复用各 L2 已定的 xDS 灰度/drain/NACK。

**风险** 发布影响在途连接 → drain + 客户端重连;客户端碎片化版本 → 协议版本兼容 + 强制升级策略(待确认);灰度监控不足 → 错误预算 + 自动回滚阈值(3.4)。

**结论** 控制面滚动、PoP 配置 xDS 灰度(ack/NACK 回退)、PoP 二进制分批+drain、客户端灰度+回滚;各层灰度+健康检查+回滚,发布不断流。

### 3.4 可观测性落地

**背景** L1 3.14 定了指标/日志/追踪/SLO 方向;需落到栈与运营。

**目标** 定可观测栈、SLO/错误预算/告警、租户可见。

**设计**
- **栈(L1 3.14 候选)**:Metrics → **VictoriaMetrics**;Logs → **ClickHouse**(选型随遥测栈最终定);Tracing → 采样(避免热路径开销);全部带 `tenant_id` + `pop` 标签。
- **关键指标**:PoP(隧道数/吞吐/解密延迟/策略评估延迟/丢包)、控制面(API 延迟/错误率/xDS 推送滞后/策略编译耗时)、撤销推送延迟(单列 SLO)、计量(计费,计费 L2)、风险/安全事件(风险引擎/安全栈)。
- **SLO + 错误预算**:数据面 99.99%、控制面 99.9%、**覆盖区**接入延迟 P95<30ms(L1 2.2;非覆盖区单列不计入);定义错误预算;阈值告警 + 值班。
- **告警**:可用性/延迟/NACK 突增/PoP 失联/xDS 滞后/撤销延迟/风险突变;分级 + 值班 + 升级。
- **租户可见**:租户管理员经控制台只见**本租户**指标/日志(L1 3.2/3.18,RLS + authz)。
- **高基数控制**:per-tenant×per-pop 标签基数 → 聚合/采样(L1 3.14)。

依据:沿用 L1 3.14 栈与 SLO;全租户打标支撑选址/计费/持续验证/SLO;租户可见受 RLS 隔离;高基数控制成本。

**风险** 高基数指标成本 → 标签基数控制 + 聚合;日志量大 → 分级保留 + 采样;计量不可采样(计费需全量,计费 L2)→ 计量路径与可观测分离。

**结论** VictoriaMetrics(指标)+ ClickHouse(日志)〔L1 3.4 候选,选型随遥测栈定〕+ 采样 Tracing,全租户打标;SLO/错误预算/分级告警/值班;租户只见自身;高基数靠聚合/采样,计量路径不采样。

### 3.5 容灾与备份落地

**背景** L1 3.15 + C7(RPO<5min/RTO<1h)。

**目标** 定控制面/数据面/密钥的备份容灾与演练。

**设计**
- **控制面**:Postgres 连续归档 + **PITR** + **跨区域流复制副本**做 DR(RPO<5min/RTO<1h,C7);ClickHouse/Redis 按需备份;配置/IaC 入 Git(可重建)。
- **数据面**:PoP **无状态**(L1 3.1)→ 无需备份;PoP 丢失 → **多 PoP 客户端切换**(L1 3.13);配置由控制面重建(fail-static 缓存,PoP L2 3.6)。
- **密钥/CA(最关键)**:**KEK 丢失=全租户数据不可恢复** → HSM 备份 + 密钥托管/恢复流程 + 双人控制(L1 3.5/3.15);CA 备份 + 应急吊销预案;国密 HSM 同理(国密选型 3.3)。
- **演练**:定期 DR 演练(故障切换、PITR 恢复、KEK 恢复)——L1 3.15"不演练=形同虚设",列常态运维项。

依据:沿用 L1 3.15 + C7;数据面无状态使 DR 主要落在控制面 + 密钥;演练常态化避免纸面 DR。

**风险** 跨区复制延迟/成本 → 按 RPO 权衡;KEK 恢复流程未演练 → 列常态演练;国密 HSM 备份/恢复 → 纳入演练。

**结论** 控制面 PITR + 跨区副本(C7);数据面靠多 PoP 切换;KEK/CA(含国密 HSM)HSM 备份 + 托管 + 双人控制;定期演练。

### 3.6 CI/CD 门禁汇总

**背景** 各 L2 散落定义了 CI 门禁(0 泄漏/RLS/策略/依赖/契约/性能/国密),需汇总成一张发布门禁表,确保"过门禁才能发"。

**目标** 汇总跨组件 CI/CD 门禁。

**设计 —— 发布门禁表(汇总各 L2)**

| 门禁 | 检查 | 来源 L2 |
|------|------|---------|
| 跨租户 0 泄漏 | 隔离测试(RLS 越权/路由域可达性=0)+ PoC-1 隔离脚本 | 数据访问层 3.9 / PoP 3.3 / L1 3.20 |
| RLS 配置 | catalog 断言(含 tenant_id 表 ENABLE+FORCE+策略;app_rw/app_ro 角色) | 数据访问层 3.7 |
| 策略正确性 | 策略 golden/冲突/原子/幂等测试 | 策略编译器 3.10 |
| 模块边界 | 依赖检查(depguard/无环)+ 不绕过 data 层 | 总览 3.3 / 数据访问层 |
| 契约一致 | proto/OpenAPI 生成物与源一致 | 总览 3.6 / 各 L2 |
| 性能 | 单 PoP 10Gbps(M1)、延迟预算、求值无解释器有界 | PoP / 编译器 / L1 3.19-3.20 |
| 国密 | (PoC-G 后)SM4/TLCP 性能、铜锁-Envoy | 国密选型 PoC-G |
| 安全 | 数据面解析器 fuzzing、依赖/SBOM 扫描、渗透 | L1 3.12/3.20 |
| 部署配置基线(PoP) | **PoP 上线前校验 `SASE_REQUIRE_CERT_TENANT=1`**(反向通道注册 fail-closed:拒绝无租户共享证书冒名注册,只认 ZTP 租户绑定证书,W9) | ZTNA 接入硬化 / SD-WAN CPE / as-built revtunnel·ZTP |

- **强制**:门禁红 → 不可合入/不可发布(L1 3.20);隔离/RLS/策略为安全门禁,不可豁免。
- **CI 门禁 vs 部署门禁**:上表前八类为 **CI 门禁**(合入/构建期);末行「部署配置基线」为**部署门禁**(PoP 实例上线前的运行时配置校验,非 CI 构建期)——dev 默认放行共享证书便于联调,**生产 PoP 必设**,否则 W9 租户绑定可被共享证书旁路。

依据:把分散在各 L2 的门禁汇成一张表,使"发布=过全部门禁"可执行;安全门禁(0 泄漏/RLS/策略)硬性。**部署配置基线**单列,因其校验对象是运行时配置而非构建产物,门禁时点在部署而非 CI。

**风险** 门禁覆盖不全=假安全 → 覆盖率纳入门禁(L1 3.20);门禁拖慢 CI → 分层(快门禁每提交、重门禁每发布)。

**结论** 八类 CI 发布门禁汇总(隔离/RLS/策略/边界/契约/性能/国密/安全)+ 一项部署配置基线(PoP require-cert-tenant),安全门禁硬性不可豁免;**国密门禁待 PoC-G 激活**(其余当前可执行);分层执行(CI 门禁合入期、部署门禁上线期)。

### 3.7 合规与多租户运营

**背景** 对外 SaaS 需合规运营 + 多租户运营(配额/计量/生命周期)。

**目标** 定合规与多租户运营要点。

**设计**
- **合规并行(R1)**:增值电信许可证 + ICP + **等保三级**取证,**与 P0 并行**(月级、上线前置,L1 3.22/R1);审计 append-only + 留存 ≥6 个月(C13,L1 3.14)。
- **多租户运营**:配额(带宽/连接器/策略数,L1 3.2)监控 + 告警;计量/计费运营(计费 L2);**租户生命周期**(开通自助/停用保数据/注销密钥销毁式删除,L1 3.16)。
- **数据合规**:SWG 解密/DLP 处理敏感数据 → 数据安全法/PIPL(安全栈 L2 3.6);数据出境(若有)单列(C1 无海外,暂无)。
- **运营可观测**:租户健康/用量/风险经控制台(L1 3.18 租户控制台)。

依据:合规是对外售卖前置(R1)、且取证月级须并行;多租户运营沿用配额/计量/生命周期(L1 3.2/3.16/3.17)。

**风险** 合规取证延误阻塞售卖 → 并行前置(R1);租户注销数据残留 → 密钥销毁式删除 + 软删除宽限(L1 3.16)。

**结论** 合规(牌照+等保并行 R1、审计留存 C13)+ 多租户运营(配额/计量/生命周期);数据合规对接数据安全法/PIPL;运营经租户控制台。

## 四、风险

### RO1:PoP 部署形态未定影响自动化/perf
缓解:倾向 systemd/裸机、PoC-1 证特权容器可行,**形态待实测拍板(PoP LO1)**;IaC 抽象使形态切换影响可控。

### RO2:发布断流
缓解:分层灰度 + drain + ack/NACK + 健康检查 + 回滚(3.3)。

### RO3:DR 演练不做=纸面容灾
缓解:DR 演练列常态运维项(3.5,L1 3.15);KEK/国密 HSM 恢复纳入演练。

### RO4:CI 门禁覆盖不全=假安全
缓解:覆盖率纳入门禁;安全门禁(0 泄漏/RLS/策略)硬性不可豁免(3.6)。

### RO5:合规取证延误阻塞上线
缓解:牌照+等保**与 P0 并行**前置(R1/L1 3.22)。

### RO6:高基数可观测成本
缓解:标签基数控制 + 聚合/采样(3.4);计量路径不采样独立(计费 L2)。

---

## 五、结论

**结论:** 运维/部署实施层——**控制面 K8s+DR、PoP IaC 一键 + N+1 + BGP 多线(形态倾向 systemd/裸机、待实测)**;**Terraform+Ansible IaC 入 Git**、平台配置走 IaC/租户配置走控制面;**分层发布**(控制面滚动 / PoP 配置 xDS 灰度 ack-NACK / 二进制分批 drain / 客户端灰度回滚)发布不断流;**可观测**(VictoriaMetrics+ClickHouse+采样 Tracing、全租户打标、SLO+错误预算+分级告警、租户只见自身);**容灾**(控制面 PITR+跨区 C7、数据面多 PoP 切换、KEK/CA HSM 备份+双人控制、定期演练);**CI 八类发布门禁汇总**(安全门禁硬性)+ **PoP 部署配置基线**(`SASE_REQUIRE_CERT_TENANT=1`,生产必设);**合规运营**(牌照+等保并行 R1、审计留存 C13、配额/计量/租户生命周期、数据安全法/PIPL)。

**衔接:** PoP L2(形态/发布/HA)、控制面总览(K8s)、xDS L2(灰度/版本)、数据访问层(PITR/迁移/RLS 门禁)、计费(计量运营)、安全栈(数据合规/SecOps)、信任风险引擎(告警)。

搭脚手架 / 写 IaC/代码须**另行授权**(设计先行);完整 runbook/脚本留实施期。
