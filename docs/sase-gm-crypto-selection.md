## 一、背景

L1 的 R7 锁定「国密必须支持」,但 L1 选型(WireGuard ChaCha20-Poly1305、标准 TLS/PKI)默认不含国密,三处链路存在算法缺口:**数据面隧道、证书体系(PKI)、SWG 的 TLS 解密链路**。其中数据面隧道是难点——WireGuard 的密码套件固定、无国密分支可直接复用。

合规档位已定为「算法级」(见前提),这放宽了选型:不要求成品过认证,可在算法实现层选择。但客群含金融/央国企,升档风险真实存在,故本文档在「算法级」做决策的同时,明确升档路径。

**为什么现在做(而非进 L2 后):** L2 的「客户端/PoP 加密栈」直接继承隧道协议与证书体系。隧道协议一旦定错,L2 客户端、PoP 数据面、PKI 签发链全部返工——返工成本随阶段递增(R7)。故须在 L1→L2 之间先定。

---

## 二、目标

1. 为**数据面隧道、PKI、SWG TLS 解密**三处分别给出国密方案:每处一个推荐 + 备选及落选理由 + 可行落地路径。
2. 满足「算法级国密」(链路实际跑 SM2/SM3/SM4),且**升档到认证级/国标 VPN 不返工**(架构预留)。
3. 守住 L1 既有约束:**不自研加密协议**(2.3)、端侧身份私钥本地生成永不离开设备(3.5)、单 PoP ≥10Gbps(2.2,国密后须按 SM4 重测)。
4. 给出**待实测项**与 P0 PoC 计划,不以未实测的性能数字下结论。

**非目标:** 不在本文档写密码学协议的完整规格(留 L2 / 选型 PoC);不覆盖国密合规的法律/测评流程(属 R1 合规路径)。

## 三、选型设计

### 3.1 国密算法与当前选型的差距映射

**背景** 先量化"缺什么",避免泛泛而谈"要支持国密"。

**目标** 量化三链路的国密算法缺口,为 3.2–3.4 的改造圈定范围。

**设计 —— 算法对应关系与各链路缺口**

| 国密算法 | 对标国际算法 | 用途 |
|---------|------------|------|
| SM2 | ECDSA / ECDH / ECIES(256 位椭圆曲线) | 签名、密钥交换、公钥加密 |
| SM3 | SHA-256 | 摘要 / HMAC / KDF |
| SM4 | AES-128(分组密码,128 位块/密钥) | 对称加密(GCM/CBC/CTR) |

| 链路 | 当前选型 | 国密缺口 |
|------|---------|---------|
| 数据面隧道 | WireGuard:Curve25519 + ChaCha20-Poly1305 + BLAKE2s(Noise_IKpsk2) | 三件套全非国密;WireGuard 套件**固定不可配**,无成熟国密分支 |
| PKI / 证书 | RSA/ECDSA 证书 + SHA-256 | 需 SM2 证书 + SM3;国密惯例为**签名+加密双证书** |
| SWG TLS 解密 | 标准 TLS 1.2/1.3(Envoy/BoringSSL) | 需国密 TLS(TLCP, GB/T 38636);BoringSSL 不含国密 |

**风险** 缺口映射若漏项(如忽略 KDF/HMAC 处的 SM3 替换)会在实现期暴露 → 由 3.2–3.4 逐链路细化 + PoC-G 验证覆盖。

**结论** 三处均需改造;隧道缺口最深(套件固定),PKI/TLS 缺口可由国密密码库(铜锁/GmSSL)填补。

### 3.2 数据面隧道选型(核心难点)

**背景** 隧道是每包热路径,且 L1 选了套件固定的 WireGuard。国密化要么改协议、要么换协议。

**目标** 选定国密隧道方案:满足算法级国密、性能可达 10Gbps 量级(待实测)、不违背"不自研加密协议"原则、升档不返工。

**设计 —— 三候选取舍**

| 候选 | 做法 | 依据(优点) | 落选/代价 |
|------|------|-----------|----------|
| **① WireGuard 国密化** | Curve25519→SM2 密钥交换、ChaCha20-Poly1305→SM4-GCM、BLAKE2s→SM3,改 Noise 框架 | 保留 WireGuard 的简洁与无连接状态模型 | **SM2 密钥交换协议与 Noise 的静态-临时 DH 模型不同构,非简单替换;无经审计的成熟实现;等于自研密码协议——直接违背 L1「不自研加密协议」原则(2.3);偏离上游、长期自维护 + 密码学审查成本** → **落选** |
| **② 国标 VPN(IPsec 国密 / GM/T 0024 SSL VPN)** | 改用 IKEv2 国密套件 或《SSL VPN 技术规范》GM/T 0024 | 合规原生、是升档到认证级的目标形态;成熟国密栈存在 | 弃 WireGuard 简洁性:IPsec 运维复杂度回归 / GM/T 0024 基于 TLS,握手与记录层开销高于 WireGuard;当前档位(算法级)不必一步到此 → **作为升档逃生通道保留(3.6),非起步首选** |
| **③ 国密 TLS(TLCP)隧道** | 数据面隧道走国密 TLS(TLCP, GB/T 38636),密码库用**铜锁(tongsuo)** | **复用 SWG(3.4)同一国密栈(铜锁)= 一套国密密码库、一处审查面**;两端我方可控,不需第三方互操作;不自研协议(TLCP 是国标);→ 升档到 ② GM/T 0024(同为 TLS 系)跳变小 | 隧道开销高于 WireGuard(TLS 握手 + 记录层);**QUIC 强绑 TLS 1.3,国密 QUIC 不成熟** → 起步用 TLCP over TCP/TLS,QUIC 国密化待验证;SM4 记录层吞吐**待实测** |

**决策:起步选 ③ 国密 TLS(TLCP)隧道(铜锁),按租户加密档位选择。**

- **依据:** ①违背不自研原则且无审计实现,直接落选——**衔接 L1 2.3:L1 将「WG 国密化」列为国密选型项(非自研新协议);本文档进一步评估其改 Noise 框架、替换 SM2 密钥交换的密码学工作量已达自研协议量级,故在选型中落选**(结论一致:均不首选);③ 相比 ② 在算法级档位下更轻、且复用 SWG 的铜锁栈(减少密码库数量与攻击面),两端可控使 TLCP 可行;② 形态重,留作升档。
- **可行落地:**
  - 隧道:端点 Agent / CPE ↔ PoP 走 **TLCP over TCP**(起步);PoP 侧用铜锁终结。
  - **按租户加密档位(crypto profile):** 国密档租户走 TLCP(SM2/SM3/SM4);明确不要求国密的租户可保留 WireGuard(性能更优)。档位由租户配置项驱动,数据面据 WG 公钥/Agent 身份选择栈(对接 **L1 3.2** 租户绑定:租户身份由 WG 公钥加密绑定确立)。
  - QUIC 数据通道(3.8 connector)国密化:**待 PoC**(国密 QUIC 不成熟,起步可对 connector 通道用 TLCP over TCP)。

**风险** TLCP 隧道吞吐低于 WireGuard,可能压低单 PoP 承载(影响 5.1 单位经济性)→ 列**待实测 M-G1**;双栈(WireGuard + TLCP)增加数据面复杂度 → 加密能力抽象为 provider 接口(3.7)收敛。

**结论** 数据面隧道起步 = **TLCP(铜锁)隧道,按租户档位选择**,WireGuard 保留给非国密租户;WG 国密化落选(自研+无审计),国标 VPN 留作升档。性能须 SM4/TLCP 实测复核(M-G1)。

> **v0.4 修正 ——形态收敛为自研 UDP 数据报隧道(以 L2 为准):** 上述「TLCP over TCP 流隧道」在 L2 详设中收敛为 **自研 crypto-agile UDP 数据报隧道**(`internal/dptunnel`):数据面用 **AEAD 数据报**(SM4-GCM↔ChaCha20-Poly1305 可插拔,对应本节双栈/按租户档位)+ 计数器重放窗口 + FEC,**握手用 TLCP/TLS1.3 导出密钥(零自研密钥协商,延续本节「不自研协议」原则——自研的只是报文封装,密码学全用标准件)**,不再用 TLS 记录层承载流量。**收益:** 正好消解本节「隧道开销高于 WireGuard(TLS 握手+记录层)」「QUIC 强绑 TLS1.3、国密 QUIC 不成熟」「队头阻塞」三处风险——数据报形态无 TCP/TLS 记录层包袱,且契合 SD-WAN 多链路/FEC/亚秒切换。**国密算法栈与 C-G1 不变。** 双栈仍按租户档位(本节)经 `dptunnel` AEAD provider 接口收敛。详见 `sase-l2-data-plane-tunnel.md` v0.2、`sase-l2-tunnel-handshake.md`(形态 A,已认可)。**SWG 解密(3.4)/管理面(3.8)仍用 TLCP 流不变。**

### 3.3 PKI 国密化(SM2 证书体系)

**背景** L1 的 3.5 定了 Root 离线 + 分用途中间 CA + 短期证书 + 信封加密。国密化主要换算法,不改层级。

**目标** 证书体系支持 SM2/SM3 签发;守住端侧私钥本地生成原则;HSM 选型兼顾升档。

**设计**

- **算法:** 各级证书签名算法 SM2 + 摘要 SM3;CA 层级(3.5)不变,仅签发链算法替换。可与国际算法(RSA/ECDSA)**并行支持**,按租户/链路档位择一签发。
- **双证书体系(国密惯例):** TLCP 端实体常用**签名证书 + 加密证书**两套密钥对。
  - **签名证书私钥:守 3.5 原则——端侧本地生成、永不离开设备。**
  - **加密证书私钥:** 国密规范允许由 KMC(密钥管理中心)托管以支持密钥恢复。算法级档位 + 端可控下,**起步坚持本地生成、不托管**(与 3.5 一致);**待确认**:若客户要求加密密钥恢复(部分金融客户),再引入 KMC 托管(升档项,3.6)。
- **CA 软件:** 基于铜锁/GmSSL 的国密 CA 能力签发 SM2 证书(与隧道/TLS 同栈)。
- **HSM:** Root CA 私钥、KEK、SWG 解密 CA 私钥仍入 HSM(3.5)。**须选支持 SM2/SM3/SM4 的国密 HSM(加密机)或软 HSM(铜锁 + PKCS#11 国密)**。**依据:国密加密机本身多已过商密认证,起步用它即为升档到认证级铺路**(逃生通道,3.6)。
- **吊销:** 延续 3.5——短 TTL + 吊销表经 xDS 推送(算法无关,不变)。

**风险** 双证书使端侧密钥管理与签发流程复杂化 → L2 客户端加密栈须显式建模签名/加密两套密钥;国密 HSM 选型不当致性能/兼容缺口 → 纳入实测(M-G2)。

**结论** PKI 层级不变,算法换 SM2/SM3,与国际算法并行、按档位择一;签名私钥守本地生成原则,加密私钥起步不托管(密钥恢复留升档);HSM 选国密型号兼顾升档。

### 3.4 TLS 解密链路国密化(SWG)

**背景** L1 的 3.10 定 SWG 选择性 TLS 解密,Envoy + 动态签发被代理站点证书。国密客群的浏览器/客户端多走 TLCP,解密链路两端都需国密。

**目标** SWG 解密/重加密链路支持 TLCP;动态签发用 SM2 证书;明确 Envoy 国密落地方式。

**设计**

- **协议:** 解密链路支持 **TLCP(GB/T 38636)**;对国密浏览器/客户端走 TLCP,对国际 TLS 站点仍走标准 TLS——**按站点/客群协商,双栈并存**。
- **动态签发:** SWG 重签的被代理站点叶子证书用 **SM2**,由 per-tenant 解密子 CA 签发(对接 3.3 + L1 3.5)。
- **Envoy 国密落地(两方案):**

| 方案 | 做法 | 取舍 |
|------|------|------|
| a. 铜锁-Envoy | 用铜锁替换 Envoy 的 BoringSSL,获得 TLCP 能力(阿里 Higress/Envoy + 铜锁 有实践) | 单代理、链路短;**代价:维护打补丁的 Envoy 构建,跟上游同步成本** |
| b. 前置国密终结代理 | 国密 TLS 终结器(铜锁/国密网关)前置,终结后明文/标准 TLS 转 Envoy | 隔离国密到独立组件、Envoy 不动;**代价:多一跳 + 一个组件,明文段需在可信边界内** |

- **决策(待 PoC):** 倾向 **a. 铜锁-Envoy**(链路短、与隧道/PKI 同用铜锁栈,一套国密库),但**铜锁-Envoy 的国密 `ext_authz`/`ext_proc` 路径稳定性与吞吐待 PoC**;若 PoC 暴露维护/性能问题,回退 b。

**风险** 铜锁-Envoy 构建与上游 Envoy 版本同步成本 → 评估锁定 LTS + 定期合并;TLCP 解密增延迟(SM2 握手)→ 对接入延迟预算(L1 3.19)实测(M-G2)。

**结论** SWG 解密支持 TLCP、SM2 动态签发、国密/国际双栈并存;Envoy 国密起步倾向铜锁-Envoy,稳定性/吞吐待 PoC,前置终结代理为回退。

### 3.5 性能与实测计划

**背景** 国密最大的未知是性能:SM4 无类 ChaCha20 的"免硬件加速即快"特性,TLCP 握手比标准 TLS 重。L1 的 10Gbps/PoP、30ms 接入延迟预算须按国密重测。**本节不下未实测的性能结论。**

**设计 —— 待实测项(M-G1/M-G2 已测并于 L1 v0.5 并入附录 B;M-G3~M-G5 部分待国密 CPU)**

| 编号 | 实测内容 | 关联 | 为何关键 |
|------|---------|------|---------|
| M-G1 | **SM4-GCM 单核吞吐**(对比 ChaCha20),在选定 CPU 上测;含是否有 SM4 硬件指令(**待确认硬件:海光/兆芯 x86、ARMv8.4、新 Xeon 是否带 SM4 指令**) | 3.2 / L1 3.7 / 2.2 | 直接决定国密下单 PoP 承载与单位经济性(L1 5.1) |
| M-G2 | **TLCP 握手延迟**(SM2 签名/验签/密钥交换)与解密链路增量延迟 | 3.4 / L1 3.19 | 是否冲击覆盖区 P95<30ms 预算 |
| M-G3 | **铜锁-Envoy 国密路径**吞吐与稳定性(ext_authz/ext_proc 同时启用) | 3.4 | 决定 Envoy 国密方案 a vs b |
| M-G4 | **TLCP-over-QUIC 可行性**(connector 数据通道国密化) | 3.2 / L1 3.8 | 决定 QUIC 通道能否国密化,还是退 TCP |

**依据:** SM4 软件实现吞吐与硬件加速强相关——有 SM4 指令时可达高吞吐,无则可能显著低于 ChaCha20(具体倍数**待实测**,不在此编造)。故硬件选型(L1 M1)须把"是否带 SM4 加速"纳入。

**结论** 国密性能为最大未知,M-G1~M-G4 必须在 P0 PoC 实测;硬件选型须考量 SM4 加速;在实测出数前,L1 的 10Gbps/30ms 对国密链路标注"按 SM4 重测"。

### 3.6 升档逃生通道(认证级 / 国标 VPN)

**背景** 合规档位现为算法级,但金融/央国企签约可能抬到认证级或要求 GM/T 0024 国标 VPN。须保证升档不返工。

**目标** 在算法级起步架构中预留升档锚点,使档位上抬为"切换实现/对齐规范"而非"重做选型"。

**设计 —— 预留三条升档路径**

1. **升到「认证级」(需过商密认证的加密机/模块):** 因 3.3 起步即选**国密 HSM/加密机**承载 Root/KEK/解密 CA 运算,且 3.7 把密码运算抽象为 provider 接口,升档=把软实现(铜锁)的 SM 运算切到已认证加密机(PCI-E 密码卡 / 网络密码机),**不改协议与证书体系**。
2. **升到「GM/T 0024 国标 SSL VPN」:** 因 3.2 起步选 TLCP(同为 TLS 系),升到 GM/T 0024 是协议规范对齐而非重写,跳变小于从 WireGuard 起步。
3. **加密密钥恢复(KMC 托管):** 3.3 已预留——起步不托管,需要时引入 KMC,只影响加密证书私钥流程,不动签名链。

**风险** 升档锚点若未被 PoC-G 覆盖(如只测软实现、未验证加密机 PKCS#11 路径),升档时仍可能现问题 → PoC-G 至少验证一条加密机路径(对接 3.7 provider 接口)。

**结论** 三条升档路径均在起步架构中预留(国密 HSM + provider 抽象 + TLCP 基线),升档为"切换实现/对齐规范",非"重做选型"。

---

### 3.7 密码能力抽象(provider 接口)

**背景** 双栈(WireGuard/TLCP)、软实现/加密机、国际/国密并行——若散落在各处,升档与维护成本失控。

**目标** 用一层接口隔离"算法实现"与"业务逻辑",使双栈共存与升档(3.6)只换实现、不改调用方。

**设计** 把密码运算(签名/验签、密钥交换、对称加解密、证书签发)抽象为 **crypto provider 接口**,实现可插拔:
- `software-intl`(BoringSSL,国际算法)
- `software-gm`(铜锁,国密算法,起步)
- `hsm-gm`(国密加密机 via PKCS#11,升档认证级)

数据面/PKI/SWG 只依赖接口,按租户档位与合规档位选实现。**依据:** 这是 L1「不自研加密协议」与本文档「升档不返工」的工程落点——provider 接口本身不实现密码学,只编排已有实现。

> 签发用途清单(供 PoC-G 范围对齐):provider 的「签名/验签」服务于 ① PKI 证书签发(3.3)② TLS/TLCP 握手(3.2/3.4)③ **会话凭证签名**(身份与 IdP 子 L2 `sase-l2-cp-identity-idp.md` 的自签短 TTL 凭证,SM2/ECDSA 敏捷)。第 ③ 项为 L2 引入的新增签发用途,**PoC-G 须一并验证**(算法/性能),勿遗漏。

**风险** 接口抽象不当致性能损失(热路径不应有多余间接层)→ 数据面对称加解密走零拷贝/直调,provider 抽象主要用于握手/签发等非每包路径。

**结论** crypto provider 接口隔离"算法实现"与"业务逻辑",支撑双栈与升档;热路径避免抽象开销。

---

### 3.8 管理面与控制信令面 TLS 国密(v0.2 补)

**背景** v0.1 只覆盖**数据面隧道**(3.2)与 **SWG 数据面解密**(3.4)两面。但系统还有一组**非数据面隧道的 TLS/mTLS 面**,强合规客户的"全链路国密"会要求它们也国密化——这些 v0.1 遗漏(前端控制台 L2 交叉审发现,原 C-G4)。

**目标** 枚举这些面、分类给国密方案,确立"我方控两端的内部面直接国密、第三方控一端的面用前置终结"。

**设计 —— 面清单与方案**

| 面 | 两端 | 国密方案 | 依据 |
|----|------|---------|------|
| 浏览器 ↔ 控制面 Admin API(管理面) | 客户浏览器(可能国密浏览器)/ 控制面 | **前置国密 TLS 终结**:国密浏览器走 TLCP 到前置网关,网关后用内部 TLS;国密/国际双栈按客户 | 浏览器侧不一定可控,与 SWG 面向任意客户端同理(借鉴 3.4b 前置终结形态,但流量路径不同) |
| PoP ↔ 控制面 xDS gRPC(mTLS) | 均我方组件 | **直接国密 mTLS**(TLCP/SM2 证书,铜锁 + provider 抽象 3.7) | 两端可控,复用国密 PKI(3.3)与铜锁栈,无需前置 |
| connector / CPE / Agent ↔ PoP 控制通道(mTLS) | 均我方组件 | **直接国密 mTLS** | 同上;凭证/证书走国密 PKI(3.3) |
| 控制面内部 mTLS(单元间) | 均我方组件 | **直接国密 mTLS** | 同上 |

- **两类处理(决策):** ① **我方控两端的内部面**(xDS、控制通道、内部 mTLS)→ **直接国密 mTLS**(TLCP/SM2,铜锁),复用 3.3 PKI 与 3.7 provider,**首选**;② **第三方控一端的面**(浏览器↔控制台)→ **前置国密 TLS 终结**(国密/国际双栈),前端/后端应用算法无关(前端控制台 L2 已据此设计)。
- 依据:内部面两端可控,直接国密最简、复用既有国密栈;管理面浏览器侧不可控,前置终结隔离国密、应用无感——与 3.4 SWG"面向任意客户端用前置终结回退"同理。
- 备选:管理面也要求所有客户用国密浏览器、不前置。落选:强加客户端约束、不现实;前置终结对客户透明。
- 备选:内部面也走前置终结。落选:内部两端可控,直接国密更简,多一跳无收益。

**风险** 国密 mTLS 面增多致性能/连接开销 → 控制信令面非每包热路径,开销可接受(待 PoC-G 一并测,M-G 扩展);前置终结成管理面单点 → 前置网关高可用(部署 L2);双栈配置复杂 → provider 抽象(3.7)统一。

**结论** 非数据面隧道的 TLS/mTLS 面补齐国密:**内部面(xDS/控制通道/内部 mTLS)直接国密 mTLS**(铜锁+SM2 PKI),**管理面(浏览器↔控制台)前置国密 TLS 终结**(双栈、应用无感);纳入 PoC-G 性能验证。

## 四、风险

### RG1:SM4 吞吐低于 ChaCha20,压低单 PoP 承载
无 SM4 硬件加速时软件 SM4 可能显著慢于 ChaCha20 → 单 PoP 承载下降、单位经济性恶化。缓解:硬件选型纳入 SM4 加速考量(M-G1);按租户档位,仅国密租户走 SM4。**倍数待实测,不编造。**

### RG2:TLCP 隧道开销高于 WireGuard
握手与记录层开销使接入延迟与吞吐劣于 WireGuard。缓解:TLCP over TCP 起步、连接复用;实测对接 30ms 预算(M-G2);非国密租户保留 WireGuard。

### RG3:铜锁-Envoy 维护与上游同步成本
打补丁的 Envoy 构建需跟上游同步。缓解:锁 LTS + 定期合并;回退前置终结代理(3.4b)。

### RG4:国密 QUIC 不成熟
QUIC 强绑 TLS 1.3,国密 QUIC 缺成熟实现。缓解:connector 通道起步 TLCP over TCP;QUIC 国密化待 PoC(M-G4)。

### RG5:升档要求突现(认证级/密钥恢复)
签约客户临时要求认证级或 KMC 密钥恢复。缓解:3.6 三条升档路径已在起步架构预留。

### RG6:双栈复杂度
WireGuard + TLCP 双栈增加数据面与测试面。缓解:crypto provider 抽象(3.7);隔离测试覆盖双栈(对接 L1 3.20)。

## 五、结论与行动项

**选型结论(算法级国密档位,待 PoC 复核性能):**

| 链路 | 结论 | 备选/逃生 |
|------|------|----------|
| 数据面隧道 | **TLCP(铜锁)隧道,按租户档位选择**;非国密租户保留 WireGuard | WG 国密化落选;国标 VPN(GM/T 0024)留升档 |
| PKI | **SM2/SM3 证书,与国际算法并行**;签名私钥本地生成,加密私钥起步不托管;国密 HSM | KMC 托管留升档 |
| SWG TLS 解密 | **TLCP + SM2 动态签发,国密/国际双栈**;Envoy 起步铜锁-Envoy(待 PoC) | 前置国密终结代理为回退 |
| 管理面 / 控制信令面 TLS(3.8,v0.2) | **内部面(xDS/控制通道/内部 mTLS)直接国密 mTLS**(铜锁+SM2 PKI);**管理面(浏览器↔控制台)前置国密 TLS 终结**(双栈、应用无感) | — |
| 升档 | 国密 HSM + crypto provider 抽象 + TLCP 基线 → 认证级/GM/T 0024/密钥恢复均不返工 | — |

**这些结论"待 PoC 复核"——性能(M-G1~M-G5)未实测前不固化进 L1 选型。**

**行动项(P0):**
1. **PoC-G:** 搭铜锁,实测 M-G1(SM4 吞吐)、M-G2(TLCP 握手延迟)、M-G3(铜锁-Envoy)、M-G4(TLCP-over-QUIC)、M-G5(控制信令面国密 mTLS 开销,3.8)。
2. **硬件选型联动:** L1 的 M1(10Gbps 实测)硬件须纳入"是否带 SM4 加速"——确认候选 CPU(海光/兆芯/ARMv8.4/新 Xeon)的 SM4 指令支持(C-G1)。
3. **回写 L1:** ✅ 隧道/PKI/TLS 国密结论已落进 `sase-architecture-design.md` 的 3.5/3.7/3.10/3.21/R7(v0.5 并入 PoC-G 结论;v0.6 并入隧道形态收敛)。余 M-G3~M-G5 带加速 SM4 数待国密 CPU 复测后增量回写。
4. **解锁 L2:** 本选型结论(+PoC)是 L2「客户端/PoP 加密栈」的前置;PoC-G 出结论后即可启动该 L2。
